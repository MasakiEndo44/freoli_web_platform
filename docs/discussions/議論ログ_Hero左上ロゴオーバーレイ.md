- 🧑‍💻 **北條（フロントエンド実装専門家）**：React / Next.js 歴 10 年、Web a11y 実務 6 年。Tailwind の z-index・position・next/image 設計が専門。

## Turn 1｜現状診断 — そもそも「Hero 上にロゴ」は何を解く問題か

**みさき**：え待って、Hero にもう「FREOLI」ってデカく書いてあるじゃないですか〜。それなのに左上にもロゴ載せるの、なんか被ってない？なんで載せるんですか？

**桐谷**：いい問いです。実は Hero の「FREOLI」文字とロゴマークは、デザイン文法上は別レイヤーの仕事をしています。Hero のタイポは "**世界観の宣言**"（暗がり・原風景・東京の四人組）、ロゴマークは "**識別子**"（このサイト = FREOLI 公式）。アー写中心のヒーローはフォトギャラリーやポスターと見分けがつきにくくなる瞬間がある。SNS からの流入経路では、ロゴが画面どこかに常駐していると「フォトギャラリーじゃない、公式サイトだ」と 0.3 秒で確定できます。私が支援した 12 件のアーティストサイトで、Hero テキストだけ・ロゴなしの構成は離脱率が平均 8% 高かった。

**北條**：技術側から補強します。`components/sections/Hero.tsx` を読みました。Hero は `<header className="relative">` で `h-[524px] lg:h-[620px]`。左上は完全に空いていて、絶対配置の競合相手がいない（テキストは bottom-0 left-0 right-0 配置）。つまり **「左上の空き地」は意図的にデザインで残されている領域**で、そこにロゴを置くのは構造的に自然です。これは「重複ブランディング問題」ではなく、**"スクロール後にもブランドが識別可能か"問題** ── ただし今回のスコープでは Hero 内 absolute に絞れば fixed/sticky の苦悩は回避できる、という診断になります。

**みさき**：あー、ロゴは "ここが公式" の名札みたいなことか〜。Hero のデカ字は "おしゃれな自己紹介" って感じ？

**桐谷**：その理解で完璧です。名札と自己紹介は両立します。

> **Turn 1 診断**：本件は「ロゴ vs Hero タイポの重複」ではなく、**「アー写中心 Hero における識別子の欠落」を埋める作業**。Hero の役割（世界観の宣言）を侵食せず、左上の空き地に "名札" を置く問題に再定義される。

## Turn 2｜選択肢の発散 — 配置・挙動の評価軸を立てる

**桐谷**：評価軸を 3 つ立てます。

| 軸 | 取りうる値 |
|---|---|
| **A. スクロール挙動** | Hero 内 absolute（一緒に流れる）/ sticky（途中まで追尾）/ fixed（常時固定） |
| **B. ロゴの存在感** | 主役級（w-20+）/ 副菜（w-12〜w-16）/ 隅っこ（w-8〜w-10） |
| **C. 機能性** | 装飾のみ / クリッカブル（`/` ホーム遷移）|

**北條**：軸の上に 3 つの具体案を配置します。

- **案 A：Hero 内 absolute（一緒に流れる）+ 副菜サイズ + ホームリンク**
  - 実装：`<LogoMark className="absolute top-3 left-3 md:top-4 md:left-4 z-20" />` を Hero 内に配置
  - メリット：実装が一番素直。Hero 内で完結し、他のセクションに影響しない
  - デメリット：スクロールで消える → 「常時識別」要件は満たさない

- **案 B：sticky（ヘッダー全体に常時追尾）**
  - 実装：layout レベルで `<header className="sticky top-0 z-50">` を新設
  - メリット：常時ブランド表示で SNS 流入時の安心感
  - デメリット：モバイル縦画面で 48px 帯が常に視界を奪う。Hero の没入感が落ちる

- **案 C：fixed + fade in/out（Hero 越えたら出現）**
  - 実装：Intersection Observer で Hero 通過後に fixed top-4 left-4 表示
  - メリット：Hero 内は邪魔せず、スクロール後はブランド常駐
  - デメリット：Client Component 化 + JS、複雑度急増。今回のスコープ過剰

**みさき**：私 Instagram から流れてきたんですけど、正直スクロールしてからもロゴ見えてた方が "あ、ここ FREOLI じゃん" って続く気がするんですよね〜。でも案 B のスマホで上に常にバー出るやつ、めっちゃ邪魔じゃないですか？画面狭くなる〜。

**桐谷**：みさきさんの直感はそのまま UX 原則になっています。**「常時表示の心理的安心 vs 視野占有のストレス」のトレードオフ**。私の経験では、独立系バンドサイトのように **滞在時間 1〜2 分の短期セッションでは、案 A（Hero 内 absolute）で十分** なケースが多い。理由は、ファンは Hero → ライブ → メンバー...と短時間で順に縦スクロールし、戻り操作は SNS タブ / ブックマークで行う。常時ヘッダーは長尺コンテンツ（ニュースサイト・SaaS）向けの装具です。

**北條**：今回のユーザー要求文も「画面の **左上はじ** にオーバーレイ表示」── 限定的な領域指定です。fixed/sticky 寄りの "常時表示" ニュアンスは弱い。**案 A が要件最小** で、案 C は将来必要になったら段階的追加可能（Hero 内 LogoMark を流用すればよい）。

> **Turn 2 暫定方向**：**案 A（Hero 内 absolute + 副菜 + クリッカブル）** を主軸。案 C は将来オプションとして据え置く。

## Turn 3｜要件・優先度 — 実需を抽象パターンに落とす

**みさき**：あと私、スマホで見ること多いから聞きたいんですけど、ロゴってタップしたらどうなって欲しいか分かります？私は普通に "ホームに戻る" だと思ったんですけど、今って 1 ページしかないですよね？

**桐谷**：そこ重要です。FREOLI サイトは現状シングルページ（`/` と `/privacy` のみ）。`/privacy` 滞在時にロゴをタップして `/` に戻れる導線は意味があります。シングルページでも「上にスクロール」をブラウザではなくロゴで誘導するのは A11y 的にも有効（フォーカス管理）。

**北條**：実需を 3 要件にパターン化します：

| ID | 要件 | 根拠 |
|---|---|---|
| **L-01** | Hero 上で識別子として機能する | Turn 1 診断 |
| **L-02** | Hero のキャッチコピー / アー写と視覚的に競合しない | AGENTS.md §1 非交渉 UX |
| **L-03** | ホーム（`/`）への遷移リンクとして機能する | みさき実需、`/privacy` からの戻り |

優先度マーカー：

- 🔴 **L-02 競合回避**（最重要、これが崩れたら non-negotiable シーケンス違反）
- 🟡 **L-01 識別子機能**（本件の目的そのもの）
- 🟢 **L-03 リンク化**（要件追加だが工数小・利便性大）

**桐谷**：L-03 は a11y 的にも「ロゴ = ホームリンク」が業界標準（WAI-ARIA Authoring Practices）。`aria-label="FREOLI ホーム"` を `<Link href="/">` に付けるだけで満たせます。コストほぼゼロ。

**みさき**：てか「競合しない」って具体的にどう？「FREOLI」ってデカ字、下の方ですよね？左上ならぶつからない気がするんですけど〜。

**北條**：そこは Hero.tsx の構造を見れば一目です。テキスト群は `bottom-0 left-0 right-0` 配置で **画面の下半分**。左上は本当に空き地。ぶつかる可能性があるのは **アー写の被写体の顔位置** で、`object-cover` で中央寄せされているので、左上端の縦 64px × 横 64px は背景グラデーション（黒フェード + 青色オーバーレイ）が支配しているエリア。**つまり L-02 は配置位置（top-4 left-4 / w-16 以下）と z-index（テキスト z-10 と被らない z-20）でほぼ自動的に守れます**。

> **Turn 3 確定要件**：L-02 競合回避（位置 + サイズ + z-index で実現）/ L-01 識別子（next/image で実装）/ L-03 ホームリンク（Link + aria-label）。3 つとも単一コンポーネントで両立可能。

## Turn 4｜技術深掘り — jpg・LCP・暗背景の三重問題

**北條**：実装の技術論点に入ります。一番厄介なのが「アセットが jpg」という点。

```
public/images/assets/freoli_logo.jpg  (49 KB, 透過なし)
```

ロゴアセットの業界標準は SVG ＞ PNG 透過 ＞＞＞ jpg。jpg を Hero（暗背景）に重ねると：

1. 白背景ロゴだった場合：白い四角が暗背景に浮いて悪目立ち
2. 黒背景ロゴだった場合：暗背景に溶けて識別不能
3. グラデーション背景ロゴだった場合：枠が見える

**桐谷**：対策案を 4 つ並べます。

| 案 | 内容 | 工数 | 仕上がり |
|---|---|---|---|
| ① **PNG/SVG 差し替え依頼** | デザイナーに透過版発注 | 高（外部依頼） | 最良 |
| ② **mix-blend-mode: screen** | CSS で白を抜く | 低 | ロゴ色が反転、白ロゴ前提でしか成立しない |
| ③ **円形マスク + 暗背景カード** | `rounded-full overflow-hidden bg-black/40` で円形カードに包む | 極小 | 元の jpg を尊重しつつ Hero に馴染む |
| ④ **そのまま貼る** | 何もしない | ゼロ | jpg の地色がそのまま見える |

**みさき**：マジ素人質問なんですけど、ロゴって透けてるのが普通じゃないですか？jpg ってそういう意味で大丈夫なやつ〜？

**桐谷**：みさきさんの直感が正解です。**ロゴアセットは透過必須が業界鉄則**。jpg は本来 NG。ただし「今すぐ載せたい」「差し替えは後日」というケースは現場で頻発する。そのとき有効なのが ③ **円形マスク + 暗背景カード方式**。ロゴ画像を中に閉じ込めるカード型 UI を作ることで、「ロゴが地色を引きずる」問題を **カードのデザインに昇華**させる。Apple Music のアーティストアイコンや Spotify のアバターと同じパターン。

**北條**：抽象的な法則として：**「アセットの欠陥は CSS のラッパーで吸収する。アセット差し替えがいつでも効くよう、画像 src は props 化する」**。これを満たすコンポーネント設計が筋。加えて LCP（Largest Contentful Paint）配慮で `priority` を入れない方がいい ── Hero 画像（freoli_1.JPEG）が既に `priority` 付きなので、ロゴまで priority にすると優先度競合します。

**桐谷**：もう一つの法則：**「Hero のコントラスト階層を壊さない」**。Hero は背景（コントラスト弱）→ オーバーレイ（中）→ テキスト（強）の 3 階層。ロゴはこの中で「中強」程度に収める。完全な高コントラスト白い箱を貼ると、視線の主役がロゴに奪われ、Hero の世界観宣言が薄まります。`bg-black/40 backdrop-blur-sm ring-1 ring-white/10` のような半透明ガラス感で中強に抑えるのが正解。

> **Turn 4 抽出法則**：
> 1. アセット欠陥は CSS ラッパーで吸収（円形マスク + 暗背景カード）
> 2. 画像 src は props 化、後日 SVG/PNG 差し替えで配線不要
> 3. priority は Hero アー写を優先、ロゴは外す
> 4. Hero のコントラスト階層において、ロゴは "中強" まで（テキストを超えない）

## Turn 5｜方針統合 — 3 原則と明日からの具体アクション

**桐谷**：設計原則を 3 つに集約します。

> 🎯 **Hero LogoMark 設計 3 原則**
>
> ① **Brand Persists, Hero Owns**
>   ブランド保持と Hero の世界観主役は両立する。ロゴは "名札"、Hero タイポは "宣言"。役割を分離すれば重複ではなく階層化になる。
>
> ② **Top-Left, Subtle**
>   左上配置・副菜サイズ（w-12 sm / w-16 md+）・コントラスト中強・Hero テキストと衝突しない z-index。攻める位置ではなく、引いた立ち位置で識別子の機能だけ果たす。
>
> ③ **Asset-Agnostic Layer**
>   jpg/png/svg いずれでも成立するラッパー設計。`src` props 化＋円形マスク＋暗背景カードで、アセット差し替え時に配線不要。

**北條**：明日からの具体アクション（実装手順）：

1. **新規コンポーネント** `components/ui/LogoMark.tsx`
   - props：`src` (default: `/images/assets/freoli_logo.jpg`)、`className`、`alt` (default: `"FREOLI ロゴ"`)
   - 構造：`<Link href="/" aria-label="FREOLI ホーム">` で `next/image` をラップ
   - スタイル：円形マスク（`rounded-full overflow-hidden`）+ 暗背景カード（`bg-black/40 backdrop-blur-sm ring-1 ring-white/10`）+ ホバー時 `scale-105`
   - サイズ：`w-12 h-12 md:w-16 md:h-16`（モバイル 48px / デスクトップ 64px）
   - `priority` は付けない（Hero アー写優先）

2. **Hero.tsx 改修** ── 1 行追加
   - `<LogoMark className="absolute top-3 left-3 md:top-4 md:left-4 z-20" />` を `<header>` 内冒頭に挿入
   - Hero の既存構造（背景画像・グラデ・タイポ）には触らない

3. **アクセシビリティチェック**
   - `aria-label="FREOLI ホーム"` で screen reader 対応
   - tap target 48px 確保（モバイル `w-12 h-12` = 48px、WCAG 2.5.5 充足）
   - alt テキストはロゴの "意味"（"FREOLI ロゴ"）であり、装飾用 alt="" にしない

4. **動作確認**
   - `pnpm typecheck` / `pnpm lint` / `pnpm build` 通過
   - dev サーバで Hero 左上にロゴ円形カード表示・タップで `/` 遷移を確認

5. **将来の差し替え経路**（記憶）
   - SVG/PNG 透過版が来たら、`public/images/assets/freoli_logo.svg` 等に配置し、`LogoMark` の default src を更新（カード装飾はそのまま残す or 解除）

**みさき**：なんか最初は「ロゴいる？被ってない？」って思ってたんですけど、"名札と自己紹介は別仕事" ってちゃんと分かりました〜。jpg のままでも円形カードでオシャレに見えるなら、後で PNG 差し替え待たなくていいし、現実的〜！てか左上にちっちゃくキラっと光るやつあると、なんか公式感マシマシで好きかも！

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| **配置** | Hero 内 `absolute top-3 left-3 md:top-4 md:left-4 z-20`（fixed/sticky なし） |
| **サイズ** | w-12 h-12（モバイル 48px）/ md:w-16 h-16（デスクトップ 64px） |
| **形状** | 円形マスク（`rounded-full overflow-hidden`） |
| **背景処理** | 半透明黒カード `bg-black/40 backdrop-blur-sm ring-1 ring-white/10`（jpg 地色を吸収） |
| **機能** | `<Link href="/" aria-label="FREOLI ホーム">` でホームリンク化 |
| **a11y** | alt="FREOLI ロゴ"、tap target 48px 確保、screen reader 対応 |
| **コンポーネント** | `components/ui/LogoMark.tsx` 新規（src props 化、後日差し替え可能） |
| **Hero 影響** | 1 行追加のみ、既存タイポ・背景・グラデは不変 |
| **priority** | 付けない（Hero アー写の LCP を優先） |
| **将来オプション** | スクロール後 fixed 表示は今回スコープ外、必要になれば LogoMark 流用で追加可能 |
