## Turn 1｜現状認識：「UI が決まらないと前に進めない」問題の正体

**森**：要件定義書を読み込みました。機能要件、データモデル、API 契約、ディレクトリ構造、ADR まで仕様レベルで固まっている。非交渉 UX シーケンス（Hero → SNSBar → NextLive → Members → Subscriptions → News → Contact）も確定済み。デザイントークン素案（cyan-400 / sky-400 × zinc-950、Noto Sans JP + Inter）まで `付録C` にある。

ここまで揃っていて何が "未確定" なのか整理すると、**「世界観の手触り」が言語化されていない**だけなんです。同じ cyan-400 でも、画面の 80% が黒で点光源として光らせるのか、ヒーロー全面にネオングラデーションを敷くのかで、印象は別物。これを Figma / Claude Design で見ないと決めようがない。

> **抽象診断**：これは「設計図はあるが、ビジュアル言語が未定義」という段階。コードを書く前に "見え方" を確定させないと、コンポーネント設計の粒度が後付けで決まってしまう。

**奥村**：森さんに完全同意です。私の経験では、Next.js を先に初期化してからデザインに戻ると、ほぼ確実に「Tailwind の utility-first に引っ張られた量産型 UI」になります。過去3案件、デザイン後付け順序でやって、`tailwind.config.ts` 再構築の手戻りで5〜7日溶けました。

5/31 公開は本日（5/17）から14日間。**手戻り1回で5日溶けたら詰む**スケジュールです。

> **抽象診断（実装側）**：「実装前のビジュアル合意が無いと、`tailwind.config.ts` と shadcn/ui の選定可逆性が落ち、後で `bg-cyan-400` を `bg-sky-500` に grep 置換するだけでは済まない構造的手戻りが発生する」

**みさき**：え待って〜！要件定義書に「ネオンブルー × 黒」って書いてあるんですよね？もう色は決まってるじゃないですか。なんで今さら6案とか作るんですか？

**森**：みさきさん、いい突っ込みです。色の **"値"** は決まっています。でも「**どの面積をどの色が占めるか**」「写真と文字のレイヤー関係」「余白の取り方」が未定。Tailwind の色コードを並べても、Hero がサカナクション寄り（観察的・余白多い）になるのか、BUMP 寄り（情緒的・文字大きい）になるのか、PEOPLE 1 寄り（ポップ・グラフィカル）になるのかで、まったく別のバンドに見える。**この方向の選択は Figma / Claude Design でしか言語化できない**んです。

## Turn 2｜選択肢の発散：Claude Design で「何を」発散させるべきか

**森**：Claude Design 発散の評価軸を3つ立てます。

| 軸 | 選択肢 |
|---|---|
| **軸A：ヒーロー構図** | (a) バンド写真フル幅 / (b) ロゴ大胆 + 写真背景小さく / (c) テキスト中心 + 写真サイド |
| **軸B：ネオン光の表現** | (a) 点光源（黒地に cyan が散る）/ (b) グラデーション（ヒーロー全面に sky→cyan）/ (c) アウトライン（テキスト・枠線だけ光る） |
| **軸C：タイポ主張** | (a) 日本語見出し中心（Noto Sans JP Bold 大胆）/ (b) 英文 "FREOLI" ロゴ中心 / (c) 混在型 |

3軸 × 2〜3パターンで **6〜9案** を発散。**選定は1案ではなく主案+バックアップ案の2案**にする。

理由：要件定義書 Phase 1 で「アー写撮影 5/24〜25、間に合わなければ既存ライブ写真10枚から選定」とある。アー写と既存ライブ写真ではヒーローの絵が変わるので、**素材リスクをデザイン段階で吸収する2案構成**が必須。

**奥村**：技術側から評価軸 D を追加します。

| 軸 | 内容 |
|---|---|
| **軸D：実装可逆性** | `tailwind.config.ts` の標準トークンだけで再現できる案か、カスタム CSS / Framer Motion が必要な案か |

これと **shadcn/ui 採否の判定** をこのタイミングで決めるべきです。私の判断は **v0.1 では shadcn/ui 不採用**。理由3つ：

1. コンポーネント数が10未満（Hero / NextLive / Members / SNSBar / Subscriptions / News / ContactForm / Footer + privacy）
2. 全部独自デザイン前提でブランド独自性が最優先（Vision「あの色のバンド、絶対観に行く」）
3. shadcn/ui を入れると Tailwind テーマが Radix 前提に縛られて、ブランド色の自由度が落ちる

**みさき**：シャドシーエヌユーアイ？なんですかそれ〜！私 Web デザイナーだけど Figma 専なので、コード側のライブラリ名は分かんない〜！

**奥村**：失礼しました。shadcn/ui は「ボタン・フォーム・モーダル等のベース実装が無料配布されている部品集」です。便利なんですが、入れるとデザインが "shadcn っぽさ" に寄ります。今回はバンドの世界観が最優先なので、入れない判断が妥当、ということです。

**みさき**：あー、なるほどね〜！便利だけど、便利すぎてみんな同じ顔になるやつね。それダサくなるやつだ〜。じゃあ、ボタンとかフォームのデザインも全部 Figma で先に決めるってこと？

**森**：その通りです。Claude Design で発散する時点で **ボタン・入力欄・フォーカス状態・エラー状態・成功状態まで含めて並べる**。実装段階で「ここ決まってない」を作らないのが鉄則。要件定義書 F3.6 の問い合わせフォームは Phase 1 必須機能なので、フォーム周りの状態デザインを初回発散から外せない。

## Turn 3｜要件・範囲の確定：みさきが持ち込む実需と、優先度のパターン化

**みさき**：私、実際 SNS からバンドの HP 見に行くんですけど、正直一番見るのは「**日付**」「**会場**」「**予約方法**」の3つだけなんですよ。写真とかロゴがどんなにおしゃれでも、それで2秒以上止まったら離脱します。逆に「あ、行けそう」って分かったら、HP の色合いとか後からじわじわ "好き〜" ってなる感じ。

**森**：みさきさんの実需、めちゃ重要です。これを抽象パターン化すると：

> **パターン：情動転換と情報取得の "二段階" を、視覚階層で同時に成立させる**
> - **第一階層（情動・0〜1秒）**：ヒーロー写真 + バンド名 = "あの色のバンドだ" の認識
> - **第二階層（情報・1〜3秒）**：次回ライブの日付 + 会場 = "行けるか？" の判定
> - **第三階層（行動・3〜5秒）**：予約・問い合わせ導線 = "予約しよう" の着地
> - North Star Quote「**手より先に心が動かされた**」の逆算が、まさにこの3階層

**奥村**：これを Claude Design 発散の優先度に落とします。

| 優先度 | 対象画面 | 発散ボリューム |
|---|---|---|
| 🔴 必須 | ヒーロー（主案+バックアップ案） | 各3案 × 2セット = **6案** |
| 🔴 必須 | NextLive セクション（日付・会場・予約の階層） | 3案 |
| 🔴 必須 | ContactForm の通常 / フォーカス / エラー / 成功状態 | 1案を4状態 |
| 🟡 推奨 | メンバーカードと hover インタラクション | 2案 |
| 🟢 任意 | ニュース・写真ギャラリーの装飾 | **発散しない**（プリミティブ流用） |

発散コストを 🔴 と 🟡 に集中させて、5/31 公開までの時間を実装に残す。

**森**：それと一点重要な配慮。**Claude Design で Figma 出力させるとき、デザイントークンの命名を `tailwind.config.ts` 直結にする**こと。Figma 上の色名を `cyan-400` `sky-400` `zinc-950` のまま使う。custom name（"FreoliBlue" 等）を作らない。これで奥村さんの実装段階で、Figma の値をそのまま `text-cyan-400` にコピペできる。

**奥村**：ありがたい配慮です。Figma → Tailwind の橋を初手で作ると、後で「Figma の色名と実装の色名が違う」事故が起きない。要件定義書付録 C のデザイントークンを Figma Variables にそのまま流し込みます。

## Turn 4｜技術深掘り：Next.js 環境セットアップの「順序の法則」

**奥村**：Next.js 環境セットアップの順序を技術側から詰めます。**6ステップ**で並べると：

```
1. pnpm + Next.js 14 App Router 初期化（pnpm create next-app）
2. Tailwind CSS インストール + tailwind.config.ts に付録C のトークンを反映
3. lib/env.ts（Zod 検証付き環境変数ラッパー）を engineering.md §2.6 通りに配置
4. engineering.md §1.3 のディレクトリ構造を空ファイルで先に作る
5. data/lives.ts / members.ts / news.ts / links.ts の Zod スキーマ + 型定義だけ先に書く
6. .claude/hooks/prevent-destructive-command.js を配置（AGENTS.md §6 Layer 1）
```

これを Claude Design の発散と **並行** で進めるのが鍵。直列にすると14日に絶対収まらない。

**森**：並行にするときの設計原則を立てます。

> **法則：デザインの未確定要素と実装の未確定要素を、同じレイヤーで衝突させない**
> - **並行可能**：環境セットアップ / データ型定義 / ディレクトリ構造 / Tailwind トークン定義
> - **並行不可（デザイン主案待ち）**：ヒーローのレイアウト実装 / メンバーカードのスタイル実装 / フォームの状態スタイル
> - これを混ぜると、デザイン主案が変わったとき「実装してから直す」が起きる

**みさき**：え、それって普通に「同時にやっていいやつ」と「ダメなやつ」を分けるってこと？じゃあ、デザイン待ってる間に裏側だけ整えていけばよくない？

**奥村**：その通りです、みさきさん。具体的にカレンダーに落とすと：

| Day | デザイン側（表側） | 実装側（裏側） |
|---|---|---|
| Day 1〜3 | Claude Design で 6〜9案を発散 | pnpm + Next.js + Tailwind + lib/env.ts + ディレクトリ + 型定義 |
| **Day 4** | **主案 + バックアップ案 選定（マイルストーン①）** | tailwind.config.ts に主案トークン反映 |
| Day 5〜7 | 5プリミティブ部品のデザイン詰め | 5プリミティブ実装（Button/Input/Card/SectionContainer/Heading） |
| **Day 7** | — | **`/styleguide` ページで5プリミティブ確認（マイルストーン②: スタイル確定）** |
| Day 8〜10 | 複合コンポーネントの調整サポート | Hero / NextLive / Members / SNSBar / Subscriptions / News / ContactForm 実装 |
| Day 11〜13 | コンテンツ・コピー流し込みレビュー | フォーム送信テスト（Resend + Turnstile 実通信）/ Vercel Analytics 設置 |
| **Day 14** | — | **5/31 本番公開** |

**森**：もう一つの法則を補強：

> **法則：「スタイル確定」とは、`tailwind.config.ts` + 5プリミティブの合意である**
> - 5プリミティブ：**Button / Input / Card / SectionContainer / Heading**
> - これが確定すれば、Hero / NextLive / Members / ContactForm 等の複合コンポーネントは「組み立て作業」になる
> - Atomic Design の Atom / Molecule の境界で発散と収束を切り分ける思想

**奥村**：補足すると、`/styleguide` ページは **公開しない内部用ページ**（`robots.txt` で除外 or 環境変数で本番非表示）にします。5プリミティブが全状態（通常 / hover / focus / disabled / error）で並んだショーケースとして、Content Owner（えんまさ）とビジュアル合意を取る場として機能させる。

## Turn 5｜方針統合：UIスタイル確定までの3原則と次の一手

**森**：要件定義書という強い制約があるからこそ、発散の範囲を絞れる。これを3原則に集約します。

### 🎯 FREOLI UIスタイル確定の3原則

**① Design Before Code, Tokens Before Components**
Claude Design でビジュアル言語を確定 → `tailwind.config.ts` に色・タイポ・スペーシングを反映 → 5プリミティブを設計 → 複合コンポーネント実装。この順序を絶対崩さない。

**② Parallel Where Possible, Serial Where Necessary**
環境セットアップ・データ型定義・ディレクトリ構造はデザイン発散と並行で進める。複合コンポーネントの実装はデザイン主案選定後の直列。混ぜない。

**③ 2 Cases, Not 1 — Photo Fallback Aware**
Claude Design は主案+バックアップ案の2案を選定。アー写撮影（5/24〜25）が遅延した場合、既存ライブ写真10枚での代替案が必要。**素材リスクをデザイン段階で吸収する**。

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
| 全体フロー | デザイン発散 → 主案選定 → Tailwind トークン化 → 5プリミティブ実装 → 複合実装、の5段階 |
| Claude Design 発散範囲 | 構図 × 光 × タイポの3軸 × 2-3案 = **6〜9案**、主案 + バックアップ案の **2案を選定** |
| Next.js 環境セットアップ | デザイン発散と **並行**。Day 1-3 で完了。pnpm + Next.js + Tailwind + lib/env.ts + ディレクトリ + 型定義 + hook |
| スタイル確定の定義 | `tailwind.config.ts` 反映済み + 5プリミティブが `/styleguide` で全状態確認済み |
| shadcn/ui 採否 | **不採用（v0.1）**。理由：ブランド独自性最優先 + コンポーネント数10未満 |
| 並行/直列の境界 | 並行可：環境・型・ディレクトリ・トークン定義 / 直列：複合コンポーネントのスタイル実装 |
| フォールバック設計 | アー写遅延に備え、既存ライブ写真10枚での代替デザイン案も並行確保 |
| マイルストーン | **Day 4 主案選定** / **Day 7 スタイル確定** / Day 14 公開 |
