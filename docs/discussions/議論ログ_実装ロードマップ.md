## Turn 2｜選択肢の発散 — ロードマップの粒度をどこに置くか

**若林**：診断は揃ったので、設計に入ります。ロードマップの粒度設計には **3 つの評価軸** があると考えます：

| 軸 | 選択肢 |
|---|---|
| **時間粒度** | (a) 週次タスクリスト / (b) 月次マイルストーン / (c) Phase（バージョン）単位 |
| **トリガー方式** | (α) カレンダー駆動（"6 月末まで"）/ (β) イベント駆動（"楽曲リリース時"）/ (γ) 指標駆動（"動員 60 人達成時"） |
| **記述形式** | (1) フラットなタスク列 / (2) Gantt 風 / (3) **依存グラフ + 入出口条件**で記述 |

このバンドの場合、**(c) Phase 単位 × (β) イベント駆動 + (γ) 指標駆動 × (3) 依存グラフ** が適切。理由は、

- 楽曲リリース日は外部要因（録音・mastering・配信代行）で動く → カレンダー固定不可
- 動員数（O-1 北極星）は遅行指標、Leading は SNS 流入と CTR → **2 段判定**
- Reversibility-First 原則上、Phase ごとに `git revert` で戻れる単位として区切るのが理にかなう

**三谷**：若林さんの 3 軸案に賛成しつつ、**1 つだけ追加**したい。**「外部依存軸」**です。FREOLI 固有事情として、

- **アー写撮影**：被写体 4 人の合意済写真が無いと Hero / Members / PhotoGallery が前進できない（CAP-004）
- **楽曲マスター**：Spotify / Apple Music への配信代行登録が完了しないと FEAT-006 Embed 化（OP-01）が発火不能
- **ブッキングオファー**：ブッカー側が動かないと v1.0 サーキット出演条件が満たされない

つまり**「コードでは終わらない、現実世界の事象に依存する Feature」**を、ロードマップで明示する必要がある。私の支援先 20 組の経験で言うと、ここを「TBD」のまま放置したロードマップは 9 割が 3 ヶ月以内に陳腐化します。

**みさき**：あー、それマジ分かる〜！私が好きなバンドも「楽曲リリース予定」って書いてあったのに半年経ってもまだ「予定」のまま、みたいなのめっちゃある！HP を見てる側からすると、**「予定」が「いつまでに変わるか」が分からないと一気に冷める**んですよ〜。「2025 年配信予定」だけだと、もう 2026 年だし、私だったら「あれ、解散した？」って思っちゃう。

**若林**：みさきさんの観察、めちゃくちゃ重要です。これは **「ロードマップは内部資料だが、UI 文言経由で外部にも漏出する」** という法則の現れですね。`data/links.ts` の `status: "coming-2025"` が 2026 年になっても残っていたら、それは Eval の ST-01（"2025 年配信予定" 表示）違反になる。

**ロードマップの選択肢を具体化**するとこうなります：

```
案A: Phase 単位の依存グラフ（v0.2 / v0.5 / v1.0 の入口・出口・Feature割当）
案B: 月次マイルストーン × 外部依存トラッキング表
案C: Backlog（やるべき小タスクの優先度付きリスト）+ Phase 概要
```

私の推奨は **案A + 案C のハイブリッド**。Phase 単位の依存グラフを骨格にして、Phase 内の具体タスクを Backlog で運用する。Gantt は避ける（日付に締め切られて Reversibility と相性が悪い）。

## Turn 3｜要件・優先度の確定 — みさきの実需から逆算する

**みさき**：私、実 Primary Persona（高橋みさき）として正直な実需を持ち込んでいいですか？私が FREOLI を Instagram で見つけて HP 開くとき、**3 秒以内に確認したい順番**は決まってて：

1. **次のライブ、いつ・どこ？**（チケット予約までいけるかも）
2. **どんな音、どんな声？**（Spotify で 30 秒聴いて判断）
3. **メンバー、誰？4 人？**（顔と名前を見て世界観を確認）
4. **最近何やってる？**（活動が止まってないか）

この **(1) → (2) の遷移** で、楽曲がまだ無いってなると、**「あ、ライブで初めて聴くしかないバンドね」**って分かって、それはそれで「ライブ行く動機になる」んですよ。だから楽曲リリース前の v0.1 期と楽曲ありの v0.5 期は、**ユーザーの心の中で全然違うバンドに見える**。ここ重要〜！

**三谷**：みさきさんの今の発言、**ロードマップの優先度設計に直接効く本質**です。整理すると：

> **v0.1 期 = "ライブで聴くバンド" として完結する設計**
> **v0.5 期 = "サブスクで予習してからライブに来るバンド" への昇格**

これは **「v0.5 への遷移は単なる Feature 追加ではなく、ユーザーが想起するバンド像の変化」**を意味する。Embed 追加（FEAT-006 OP-01）が表面的なタスクだけど、その裏で **「みさきさんが SNS で見つけた瞬間、サブスクで予習する」という新しい行動経路が開く**。だから v0.5 では:

- **(a) サブスク Embed 表示**（OP-01、表面）
- **(b) Hero コピー文の刷新**（"ライブで会いに行くバンド" → "聴いて、観に行くバンド"）
- **(c) News 投稿のリリース告知運用**（FEAT-007）
- **(d) Spotify アーティストページ整備**（外部・並行）

の 4 つが**同時に着地する必要がある**。Embed だけ追加しても、Hero コピーが古いままだと UX が割れる。

**若林**：技術側からこれを **抽象パターン化** すると：「**Phase 遷移＝Feature の同時集中変更点**」という法則になります。Reversibility-First の観点で言うと、

> **1 つの Phase 遷移は 1 つの PR で完結すべき** ── 4 ファイル横断する PR でも 1 つに束ねる

これは AGENTS.md §8「ブランチ管理規則」と整合する。`content/v05-subscription-embed-launch` のような単一ブランチで Phase 遷移を行えば、不具合時に Vercel revert で **その Phase まるごと前バージョンに戻せる**。これが Reversibility-First の真の意味です。

具体的に Phase 1 残タスク / v0.2 / v0.5 / v1.0 の優先度を決めましょう：

| 優先度 | Phase | 必須タスク | 同時着地が必要なもの |
|---|---|---|---|
| 🔴 最優先 | **Phase 1 残** | FEAT-008 ContactForm の API ルート実装（Resend + Turnstile） | `app/api/contact/route.ts` + サーバー検証 + 503 fallback の動作確認 |
| 🔴 最優先 | **Phase 1 残** | FEAT-009 Privacy + FEAT-012 CONTENT_POLICY.md 配置 | 法的素案レビュー + リポジトリ直下配置 |
| 🔴 最優先 | **Phase 1 残** | Members 3 名分（ゆうすけ/ひろむ/aberyo）の bio + 写真 + 承諾 | Notion Yes ログ整備 |
| 🟡 次点 | **v0.2** | FEAT-010 PastLives（`actual_attendance` 込み、D-O1/D-F2 実装） | 7/11 含む過去ライブ実測値の遡及記入 |
| 🟡 次点 | **v0.2** | FEAT-004 PhotoGallery（被写体本人 Yes ログ照合） | CONTENT_POLICY.md の運用回路の最初の発火 |
| 🟢 中期 | **v0.5** | サブスク Embed + Hero コピー刷新 + News + Spotify アーティストページ（4 つ同時着地） | 配信代行登録完了がトリガー |
| 🟢 中期 | **v1.0** | 独自ドメイン取得 + 301 リダイレクト + プレスキット + メーリングリスト検討 | サーキット出演オファー受領がトリガー |

**みさき**：あー、この優先度マジ納得〜！特に v0.2 で PastLives 入れるってやつ、私的にめっちゃ大事です。**「このバンド、ちゃんとライブやってきてるんだ」**って分かると、信頼度爆上がりするから。あと **動員数が出てると「人気度が一目で分かる」**から、ブッカーの田村さんが 5 分で判断できる、っていうやつの根拠資料になりますよね？

**三谷**：その通り。みさきさんが Persona の Tier 1 シグナル「ブッカーから動員力を確認した上でオファー」を引き出してくれた。CAP-006 × FEAT-010 × `actual_attendance` の三位一体です。

## Turn 4｜技術深掘りとリスク — Reversibility と権利ガバナンスの法則

**若林**：技術側でロードマップに織り込まなきゃいけない **個別ノウハウ** を 5 つ挙げます。

1. **Vercel ビルド時間の上限**：Hobby プランは月 100 GB-hours の制限。画像最適化を `next/image` 任せにすると Phase 進行で容量が膨らむ。FEAT-004 PhotoGallery 追加時は **画像圧縮 (WebP/AVIF) のソース側事前最適化** が必須。
2. **環境変数の追加プロセス**：v0.5 で `SPOTIFY_ARTIST_ID` `APPLE_MUSIC_ARTIST_ID` のような環境変数追加が発生する可能性。**ENV 追加は必ず `lib/env.ts` の Zod スキーマも同時更新**。これを怠ると build 時 fail-fast が緩む。
3. **Resend ドメイン検証**：v1.0 独自ドメイン取得時、Resend 側でも SPF/DKIM 設定が必要。**ドメイン切替 PR と Resend 設定変更は同じ日に実行できない**（DNS 浸透に 24h かかる）。
4. **Turnstile サイトキーは静的**：独自ドメイン取得時、Cloudflare Turnstile 側でドメイン追加登録が必要。新ドメインで一時的に Turnstile が機能しなくなるリスク → 旧ドメインを 1 週間並行運用してから cutover する。
5. **Vercel revert の盲点**：環境変数変更は `git revert` で戻らない。ENV 変更を伴う PR は **revert 不可能な変更**として PR 説明で明示する規約が要る。

これらをロードマップ上の **「Phase 遷移時の同時実行チェックリスト」** として言語化する必要があります。

**三谷**：若林さんが個別ノウハウを 5 つ挙げてくれたので、私が **支配する法則** を抽出します：

> **「Phase 遷移時の不可逆要素（DNS / ENV / 外部 API 鍵 / 写真の公開）は、コードの可逆要素（UI / コピー / レイアウト）から分離してロードマップに記載する」**

これを **可逆 / 不可逆の分離原則** と呼びます。具体的には、ロードマップに 2 つの列を持つ：

| 可逆タスク（git revert で戻る） | 不可逆タスク（戻らない or 戻すのに数日かかる） |
|---|---|
| UI レイアウト変更 | DNS / ドメイン切替 |
| `data/*.ts` 更新 | 環境変数追加・変更 |
| Hero コピー文 | Resend / Turnstile 側の設定 |
| News 投稿 | 公開済み写真の削除依頼（被写体合意の取り直し） |
| Tailwind スタイル | Spotify アーティストページの作成 |

**不可逆タスクは Phase の "頭" に集中**させ、可逆タスクで Phase 内の細かい調整を吸収する。これが Reversibility-First を Phase レベルで実装する方法です。

**みさき**：え待って、「不可逆」って言葉、私的にはめっちゃ刺さるんですけど、**普通の人にこれを毎回判定させるのって難しくないですか？** 私が FREOLI のメンバーだったら、「これ revert で戻るやつだっけ？」って毎回考えるの絶対忘れる〜。

**若林**：みさきさんの指摘、これは **運用ガードレール設計** の話に直結します。具体策として：

1. **PR テンプレートにチェックボックス**：「この PR は不可逆要素を含む？ □ Yes（要 Tech Owner 承認） □ No」
2. **`.github/PULL_REQUEST_TEMPLATE.md` でロードマップへのリンクを必須化**：「この変更はロードマップのどの Phase に該当？」
3. **`docs/operations/PHASE_TRANSITIONS.md` を新設**：Phase 遷移チェックリストを文書化（ENV 確認 / DNS 浸透 / 旧ドメイン並行運用 / 写真承諾再確認）

これらは AGENTS.md §7 Gate 1 / Gate 2 に追加すべき運用です。

**三谷**：法則として言語化すると、**「ガードレールは人間の記憶力に頼らない。テンプレートと自動化に埋め込む」**。これは Reversibility-First の運用面での具体化です。

## Turn 5｜方針統合 — FREOLI Implementation Roadmap 3 原則 と次の一手

**三谷**：5 ターンの議論を集約します。FREOLI Web Platform 実装ロードマップの **3 原則** はこれです：

### 🎯 3 原則：FREOLI Roadmap Principles

> **① Phase as Single PR — Phase 遷移は単一 PR で束ねる**
> 1 つの Phase は 1 つのブランチ・1 つの PR で完結する。これにより Vercel revert で Phase まるごと前バージョンに戻せる。

> **② Reversible First, Irreversible Front-loaded — 不可逆タスクは Phase の頭に**
> DNS / ENV / 外部 API 設定 / 公開済み写真は不可逆。可逆な UI / コピー / `data/*.ts` 更新と分離してロードマップに記載する。

> **③ Bandscape Co-evolution — バンド像の進化と HP の進化を一致させる**
> v0.5 への遷移は Embed 追加ではなく "ライブで会うバンド → 聴いて観に行くバンド" への自己像更新。Feature / Copy / News / 外部資産（Spotify アーティストページ）が同時に着地する。

### 📅 明日からの具体アクション 5 つ

**若林**：原則を打ち上げたので、明日からのアクションに着地させます。

1. **【今週中】Phase 1 残タスクの完了**：FEAT-008 API ルート実装 + FEAT-009 Privacy ページ + FEAT-012 CONTENT_POLICY.md 配置を 1 PR で束ねて launch する。
2. **【今週中】`docs/roadmap/IMPLEMENTATION_ROADMAP.md` の新設**：今回の議論を Phase 依存グラフ + 不可逆/可逆分離表として文書化。
3. **【今月中】Members 残 3 名分のコンテンツ揃え**：ゆうすけ / ひろむ / aberyo の bio + 写真 + 承諾を取得し、Notion Yes ログを整備。
4. **【来月中】v0.2 PR 計画書作成**：FEAT-004 PhotoGallery + FEAT-010 PastLives を 1 PR で着地させる計画書。`actual_attendance` の遡及記入も含める。
5. **【次の四半期】`.github/PULL_REQUEST_TEMPLATE.md` 整備**：不可逆要素チェックボックス + ロードマップ Phase 参照リンクを必須化。

**みさき**：なんか最初は「ロードマップって、Gantt チャートみたいなやつ作るんかな〜」って思ってたんですけど、議論を聞いてたら、**FREOLI の場合は時間軸じゃなくて "Phase（バンドの自己像）の進化軸"で考える**ってことなんですね〜！それマジで Lit.Link 系の汎用ページじゃできないやつだし、**「決断の最終確認所」っていう Vision とちゃんと繋がってる**感じがして、めっちゃ納得〜！あと、不可逆タスクを Phase の頭に置くって発想、私が好きなバンドの HP も全部それで運営してくれたら絶対消えないバンドになる気がする〜！

## ✅ 議論まとめ

| 項目 | 方針 |
|---|---|
