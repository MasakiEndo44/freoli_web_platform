**日付**: 2026-05-19
**スキル**: expert-misaki-discussion (Classic Mode)
**目的**: `body::before` で実装したリキッドテクスチャ背景画像が DOM では適用されているのに、ユーザー目視で「ただの黒背景」に見える原因を究明し、修正方針を確定する

## Turn 1｜現状診断：「実装は正しいが知覚されない」問題

**栗原**：DOM 上は完璧です。`body::before { background-image: url(...liquid-bg.png); opacity: 0.6; filter: blur(4px); z-index: -1; }` で computed style は全部仕様通り、画像も HTTP 200 で 686KB 配信されている。私の検証スクショ（freoli-bg-verify.png）でも Hero 上部に青いグラデーションが見える。
ところがユーザーが添えた目視スクショでは「ただの黒」。**実装と知覚の乖離**が発生しています。
ここで疑うべきは 3 点：①`body` に `bg-black` が当たっていて `body::before` の `z-index: -1` が **black の塗り潰しの下に潜っている**、②私のスクショで見えていた青みは Hero セクション内の `Hero.tsx` のグラデ wash overlay（spec §1.2 の B-b パターン）であって背景画像ではない、③画像が暗すぎる。**結局これは「z-index: -1 と body の background-color の競合が、画像をユーザーから隠している」問題**だと診断します。

**長谷川**：栗原さんの技術診断に加えて、私はもっと根本的な問題を指摘します。**「Atmosphere over Detail」原則は、ユーザーの知覚に届かなければ Atmosphere ですらない**。design_spec §1.3 で「深夜のライブハウスの空気感」と書いたのに、みさきさんが「ただの黒」と感じた瞬間に、その原則は破綻している。
私の経験では、ブランドビジュアルが「ある」と「ない」の中間は存在しません。極論「99% 透明な絵 = 透明な絵 = 何もない」。ユーザーは「微かに何かがあるかも？」とは思いません。**結局これは「Atmosphere は閾値ベースの知覚現象であって、線形ではない」問題**です。spec の opacity 0.6 / blur 4px が閾値を下回ったから、表現として 0 になっている。

**みさき**：え待って、要するに、背景の画像って **「ある」けど「ない」**ってこと？ なんかコードでは「使ってる」って表示されてるけど、私が見てる画面では「使ってない」のと同じ、っていう。なんかそれめっちゃ哲学的じゃないですか〜笑。
でも正直、私のスマホで見たやつは本当に「黒い壁紙」って感じで、ライブハウスとか深夜とか、そういう雰囲気は **マジでゼロ** でした。SNS で FREOLI 知って、サイト来て「あれ、思ったよりただのサイトじゃん」ってなったレベル。

**栗原**：みさきさんの「ある」けど「ない」、まさにそれです。CSS 的には「ある」、視覚的には「ない」。これは技術者として最も避けたい状態 — 動いているのにユーザーに価値が届いていない。

| 観点 | 状態 |
|---|---|
| 実装 (DOM / computed style) | ✅ 完全に動作 |
| 配信 (HTTP / 画像ファイル) | ✅ 686 KB 正常配信 |
| ユーザー知覚 | ❌ **ゼロ**（「ただの黒」） |
| ブランド表現 | ❌ Atmosphere 閾値以下で 0 と等価 |

## Turn 2｜原因の発散：4 つの仮説を並列で潰す

**長谷川**：原因究明の評価軸を 4 つ立てます。**それぞれ独立に修正可能**なので、優先度を付けて潰しに行きましょう。

| 仮説 | 内容 | 確認方法 |
|---|---|---|
| **H1**: z-index 隠蔽 | `body { bg-black }` の塗り潰しが `body::before { z-index: -1 }` を覆っている | body の bg-black を一時的に外してテスト |
| **H2**: opacity 過少 | `opacity: 0.6` でも画像が暗すぎて black と区別つかない | 一時的に 1.0 にしてテスト |
| **H3**: blur 過多 | `blur(4px)` で輪郭がボヤけてのっぺりした黒に見える | 一時的に `blur(0)` でテスト |
| **H4**: 画像コントラスト不足 | 686KB の画像自体が暗すぎる / コントラスト低い | 画像のヒストグラムを確認 |

**栗原**：技術的優先度は **H1 > H2 > H3 > H4** です。理由：CSS の stacking context 仕様で、`body::before { z-index: -1 }` は **body の background-color の後ろに行く**のがブラウザの一般的な挙動。これは MDN ドキュメントにも記載があります。
具体的には：
```css
/* 現在の問題実装 */
body {
  background-color: black;  /* これがオーバーレイ */
}
body::before {
  z-index: -1;  /* body の bg の下に潜る */
  background-image: url(...);
}
```

`body { background-color: black }` が **不透明な黒の塗り**として `body::before` を完全に覆っている可能性が極めて高い。私のスクショで青みが見えていたのは、Hero.tsx 内の半透明グラデ overlay（`bg-[linear-gradient(148deg,rgba(56,189,248,0.22)...)]`）であって、body::before ではなかった、と訂正します。

**みさき**：てか、その仮説 H1 が本当だったら、私の目視は **正しかった** ってこと？「画像はあるけど黒い壁紙が上に重なってて見えない」状態？ それなら原因シンプルじゃないですか。**body の bg-black を外せばいい**って話？

**長谷川**：みさきさんの直感、正解の可能性が高い。ただし、bg-black を完全に外すと「画像の opacity 0.6 の透けた部分」から **ブラウザのデフォルト白背景**が透ける可能性もある。安全策は **body の塗りを `bg-transparent` にして、html 側に bg-black を移す**、または **body::before の z-index を 0 にして body の bg の前に置く + body::before に追加で `mix-blend-mode` を使う**。
私の選好は前者：**html bg-black, body transparent, body::before のフルカバー画像**。これが最も予測可能。

**みさき**：「html に黒、body は透明、ふわっと画像を浮かせる」みたいな順番ですよね？ なんかケーキの層みたいでわかりやすい。

**結論（仮説優先度）**：H1（z-index 隠蔽）が最有力。次ターンで具体修正案を出す。

## Turn 3｜要件確定：みさきの実需 → 抽象パターン → 修正優先度

**みさき**：要件の本音を整理させてください。

1. サイトを開いた瞬間「**あ、世界観あるな**」って 1 秒で感じたい
2. ただし、「派手すぎ」は嫌（FREOLI のクールさが消える）
3. **テキストは絶対に読めること**（コントラスト 4.5:1 死守）
4. **メンバー写真とか乗ったときに邪魔しない**こと（PR #7 で Card に半透明適用したのは正解だと思う）
5. 「Hero 上部だけ青い」じゃなくて、**スクロールしても背景の気配がある**こと

**長谷川**：みさきさんの 5 点を抽象化すると、要件原則は「**Atmosphere は Continuous で Subtle、しかし Detectable**」。3 要素を分けて評価軸にできます：

| 軸 | 現状 | 望ましい状態 |
|---|---|---|
| **Continuous（連続性）** | Hero 上部のみ青み | 全セクションで気配が連続 |
| **Subtle（控えめ）** | ✅ opacity 0.6 / blur 4px は適切な強度 | 維持 |
| **Detectable（検知可能）** | ❌ ゼロ（body bg-black 隠蔽） | 「あ、画像あるな」と 1 秒で気付ける |

**栗原**：これを修正アクションに落とすと優先度が出ます。

| # | 修正 | 優先 | 根拠 |
|---|---|---|---|
| 1 | `body { bg-black }` → `html { bg-black } + body { bg-transparent }` | 🔴 必須 | H1 仮説の解消。検知可能性を確保 |
| 2 | `body::before` の z-index を再考（-1 か 0 か） | 🔴 必須 | #1 と合わせて stacking context を確定 |
| 3 | opacity 0.6 → 0.7 or 0.8 に上げる選択肢 | 🟡 検討 | みさきの「派手すぎ NG」と相反するので慎重に |
| 4 | 各セクションの背景処理を確認（Hero / Lives / etc が bg-black 当てているか） | 🟡 推奨 | #1 が効いても各 section で再度遮蔽されると意味なし |
| 5 | `design_spec.md` §1.3 を Detectable 観点でリライト | 🟢 任意 | 設計意図のドキュメント化 |

**みさき**：あ、ちょっと聞いていいですか？ Hero とかメンバーセクションが個別に `bg-black` 持ってたら、結局そこでも画像隠れちゃうんですよね？ それも直さなきゃダメじゃないですか？

**長谷川**：みさきさんの指摘、鋭い。**「html だけ黒、他は全部透明」を徹底しないと、せっかく root で透けても section で再度隠蔽される**。これがレイヤード透過の落とし穴。次ターンで実コードを当たります。

## Turn 4｜技術深掘り：実装ノウハウ → 法則と制約

**栗原**：実装ノウハウを 4 つ提示します。

1. **CSS 階層の正しい設計**：
```css
html { background-color: #000; }  /* 真の塗り */
body {
  background: transparent;  /* 透明 */
  position: relative;
}
body::before {
  content: '';
  position: fixed;
  inset: 0;
  background-image: url('/images/background/liquid-bg.png');
  background-size: cover;
  opacity: 0.7;  /* 0.6 → 0.7 微調整 */
  filter: blur(4px);
  z-index: 0;  /* -1 → 0、body の中身の下、body bg の上 */
  pointer-events: none;
}
```

2. **z-index = 0 + pointer-events: none の組み合わせ**：これで擬似要素が `<main>` の中身よりは後ろだが、body の塗り（transparent）よりは前に出る。stacking context が安定。

3. **個別セクションの遮蔽防止**：各 section の `className` から `bg-black` を外す or `bg-transparent` に統一。ただし、可読性のため必要に応じて `bg-black/40` のような半透明オーバーレイで個別調整。

4. **frontend-design skill の "Layered transparencies" 原則と整合**：背景画像（layer 1）→ body の transparent（layer 2）→ section の半透明（layer 3）→ Card の `bg-zinc-900/80 backdrop-blur-sm`（layer 4）→ コンテンツ。**4 層レイヤード**が完成。

**長谷川**：個別セクションを実コードで確認しましたが、`components/sections/Hero.tsx` には背景画像のないグラデ overlay のみ、`LivesSection.tsx` `MembersSection.tsx` `SubscribeBar.tsx` `NewsList.tsx` `ContactForm.tsx` には個別の `bg-black` がない（SectionContainer もデフォルトで背景指定なし）はず。これは PR #5 の実装時点で「body bg-black に任せる」設計だったので、**body bg-black を外せば全セクションで自動的に背景画像が透ける**。
SNSBar.tsx だけ `bg-zinc-950` を持っている。これは spec §1.2 の指示通りで、SNS バーは「個性のあるラインとして強調される」役割なので維持して OK。

**みさき**：ちょっと待って、`html` と `body` ってなんで分けて考えるんですか？ 普段「body の背景色」って言うじゃないですか〜。

**栗原**：いい質問です。HTML には **「ルート要素（html）と body の両方が viewport の塗りに関与する」** という独特の仕様があります。普通は body だけ気にすれば動くのですが、`body::before` のような擬似要素で **stacking context** を組むときは、html を「絶対的な背面」として使い、body を「中間レイヤー」、body::before を「画像レイヤー」と分けるのが堅牢な手法です。
要するに：**html = 安全網（fallback 色）、body = 透明な空気、body::before = 主役の画像**。

**栗原 & 長谷川**：これらを抽象化すると、3 つの法則が出ます。

- **法則①「Atmosphere is Threshold-Based」**：Atmosphere は線形ではなく閾値ベース。閾値を下回ったら表現として 0 になる
- **法則②「Stacking Context is a Layered Contract」**：CSS のレイヤーは契約であり、いずれかの層が contract を破ると（不透明な塗りで上書きすると）全層が破綻する
- **法則③「Visible by Default, Restrained by Choice」**：見えるが基本、控えめにするのは設計判断。最初から見えないものは存在しないのと同じ

## Turn 5｜方針統合：設計原則 + 明日からの具体アクション

**長谷川**：設計原則を 3 つにまとめます。

> ### ① **Visible by Default, Restrained by Choice**
> 背景は最初から「見える」ことを保証し、調整は opacity / blur で控えめにする。**「見えない可能性」を含む実装は採用しない**。

> ### ② **Stacking Context as Layered Contract**
> `html`（fallback 色）/ `body`（transparent 空気）/ `body::before`（主役画像）/ `section`（透明 or 半透明）/ `Card`（半透明グラスモーフィズム）の **4 層契約** を全コンポーネントで守る。

> ### ③ **Continuous Atmosphere, Section-Local Overlay**
> 背景は全画面 fixed で連続させ、可読性が必要な部分は **section レベルの半透明オーバーレイ**（`bg-black/40` 等）で個別調整する。section ごとに `bg-black` 100% を当てると遮蔽される。

**栗原**：原則を打ち上げたので具体に降ろします。**明日から取る具体アクション 5 ステップ**：

```
1. ブランチ作成: fix/v01-background-visibility
2. app/globals.css の修正:
   - html { @apply bg-black; }  ← 追加
   - body の bg-black を削除（transparent デフォルト）
   - body::before の z-index: -1 → 0 に変更
   - opacity: 0.6 → 0.7 微調整（みさき要望「世界観 1 秒検知」のため）
3. 動作確認: Mobile 375px / Desktop 1024px で目視
   - 背景画像が全セクションで気配として連続するか
   - テキストコントラスト 4.5:1 維持されているか
4. design_spec.md §1.3 更新:
   - 「stacking 4 層契約」を明文化
   - opacity 0.7 / z-index 0 を正本値として記録
5. PR 作成 → マージ
```

**みさき**：なんか最初は「画像が背景に反映されてるはずなのに見えない」って結構不思議な状況だったんですけど、要するに「**body の黒塗りが画像を覆い隠してた**」ってことですよね。それを「**html に黒を移して body を透明にする**」だけで直る、ってめっちゃシンプル！
あと「世界観 1 秒で検知」って私が言ったやつ、めっちゃ大事にしてもらえて嬉しいです〜。Spec に書いた「Atmosphere」って言葉が、技術と知覚で意味ズレてたっていうのが、議論で言語化されたのが一番収穫でした。
