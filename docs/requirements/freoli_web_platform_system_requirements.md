# FREOLI Web Platform システム要件定義書 v3（確定版）

> 「決断の最終確認所」として機能する、バンドFREOLIの公式Webプラットフォーム
> 7/11 Blue Sheep ライブの集客装置として **2025/5/31 最短公開** を目指す v0.1 を起点に、楽曲リリース・サーキット出演に同期して段階拡張する

---

## システム概要

FREOLI Web Platform は、東京・関東を拠点とする4人組インディーロックバンド「FREOLI」の公式Webサイトである。本サイトの戦略的役割は「**新規リスナー発見の入口**」ではなく「**決断の最終確認所**」── すなわち SNS や口コミで FREOLI に興味を持った人物が、**ライブ参加 と 楽曲再生** という二大行動への転換を受ける場として設計される。

楽曲未完成かつボーカル動画もSNS掲載に止める現フェーズにおいて、v0.1 のヒーロー領域はバンド写真とライブ告知を主役に据え、ネオン系の鮮やかな青（cyan/sky-400 系）×黒のトーンで FREOLI の世界観を伝える**最小構成のリンクツリー型＋ポートフォリオ型ハイブリッド**を採る。技術スタックは Next.js 14 + Tailwind CSS + Vercel Hobby + Resend で構築し、`v0.1（2025/5/31）→ v0.5（楽曲リリース時）→ v1.0（サーキット出演確定時に独自ドメイン取得）` の3段階で進化させる。問い合わせフォーム（送信先: `freoli.official@gmail.com`）を v0.1 から実装することで、ブッキング窓口としての機能も最初から備える。

## 機能要件

### F1: ライブ集客機能（最優先カテゴリ）

| ID | 要件 | 優先度 | 詳細 |
|----|------|--------|------|
| F1.1 | 直近ライブ情報の常時表示 | 必須 | トップページのファーストビュー直下に、次回ライブの日付・会場名・開場/開演時刻・チケット価格を表示。複数公演がある場合は最も近い日付を最上位に表示 |
| F1.2 | チケット予約・問い合わせ導線 | 必須 | チケット予約URL未取得を前提に、「**会場へお問い合わせください**」テキストと会場の電話番号 or 公式問い合わせフォームへのリンクを表示。将来URLが取得できれば置換可能な構造にしておく |
| F1.3 | 会場アクセス情報 | 必須 | 会場名から Google Maps へのリンク。最寄駅と所要時間をテキスト併記（例: 下北沢駅 徒歩3分） |
| F1.4 | 過去ライブ履歴 | 重要 | 過去出演履歴を日付降順で5〜10件表示。会場名・対バン・自主企画かブッキングかを区別 |
| F1.5 | ライブ予定の未定状態UI | 重要 | 次回ライブ未定時に「次回公演調整中」のプレースホルダーを表示（過去日付が残り続けるのを防ぐ） |
| F1.6 | ライブ動員数の継続記録 | 重要 | `data/lives.ts` に `actual_attendance: number \| null` フィールドを追加し、Content Owner（えんまさ）が公演翌日に実測値を記入する運用。北極星指標「全ライブの総動員数（KGI）」の計測元データ。未記入は `null` 許容で KGI 算出から除外（2026-05-18 grill 派生要件 = 設計決定 #22） |

### F2: アーティストプレゼンス機能

| ID | 要件 | 優先度 | 詳細 |
|----|------|--------|------|
| F2.1 | ヒーロービジュアル | 必須 | トップ最上部にバンド写真（メンバー4人集合 or アー写）を全幅で配置。バンド名（FREOLI のロゴまたはタイポグラフィ）と1行キャッチコピーを重ねる。ネオンブルーのアクセントカラーを差し色として使用。素材選定: 5/24〜25 撮影予定のアー写を優先、間に合わなければ既存ライブ写真10枚から選定 |
| F2.2 | バンド一言紹介 | 必須 | 「東京の4人組インディーロックバンド」レベルの簡潔な紹介文（30〜80字）。BUMP/サカナクション系統の音楽性が伝わる表現 |
| F2.3 | メンバープロフィール | 必須 | 4人のメンバー名・パート・顔写真を一覧表示。可能であれば1〜2行の個別紹介。表記順は Vo/Gt → Gt → Ba → Dr の楽器順、または年齢順を採用 |
| F2.4 | バンド写真ギャラリー | 重要 | アー写・ライブ写真を3〜5枚掲載。next/image による最適化必須。素材10枚から厳選 |

### F4: コンテンツ更新／ニュース機能

| ID | 要件 | 優先度 | 詳細 |
|----|------|--------|------|
| F4.1 | お知らせ／ニュース欄 | 必須 | ライブ告知・楽曲リリース・メディア出演等を時系列で表示。v0.1 では TypeScript ファイル内に直書きで管理。月1回以上の更新を運用目標とする |
| F4.2 | ニュース投稿のレビュー手順 | 必須 | 投稿前にえんまさ（更新担当）によるレビューを必須化。Git ベース運用時は PR レビュー、CMS 移行後はステータス管理で代替 |
| F4.3 | CMS連携（中長期） | 任意 | v0.5以降、microCMS または Notion API でメンバー自身が更新可能な仕組みに移行検討 |

### F5: 運用・ガバナンス機能

| ID | 要件 | 優先度 | 詳細 |
|----|------|--------|------|
| F5.1 | コンテンツ削除リクエスト対応 | 必須 | メンバー本人から削除依頼があった場合、1営業日以内に該当コンテンツを削除できる構造（Git管理＋自動デプロイで実現） |
| F5.2 | 権利ポリシーの明文化 | 必須 | 写真・動画・楽曲の利用ルールを `CONTENT_POLICY.md` としてリポジトリに含める |
| F5.3 | 更新担当ロール | 必須 | **えんまさが全コンテンツの更新を一括管理**。各メンバーは情報提供と承認を担当（写真の本人合意、ライブ情報の確認等） |
| F5.4 | 担当不在時のフォールバック | 重要 | えんまさが3週間以上対応不可になる場合に備え、Git リポジトリへの最低1名（推奨: aberyo or ひろむ）のアクセス権を付与しておく |
| F5.5 | プライバシーポリシー | 必須 | F3.6 問い合わせフォームで取得する個人情報（氏名・メアド）の利用目的・保持期間・削除手順を `/privacy` ページに明記 |

> **優先度の凡例**: 必須 / 重要 / 任意

## 非機能要件

| カテゴリ | 要件 | 目標値 | 根拠 |
|---------|------|--------|------|
| パフォーマンス | First Contentful Paint (FCP) | 1.5秒以内（4G環境） | SNS流入が90%以上を占める前提、モバイル遅延でユーザーが離脱する閾値 |
| パフォーマンス | Largest Contentful Paint (LCP) | 2.5秒以内 | Google Core Web Vitals の "Good" 基準 |
| パフォーマンス | ライブ情報の更新反映 | コミット後3分以内に本番反映 | Vercel の自動デプロイ標準で達成可能 |
| ユーザビリティ | モバイル表示の最適化 | 全機能がスマホで完結 | 想定流入経路（SNSプロフィールのリンク）はモバイル90%以上 |
| ユーザビリティ | ライブ集客導線の到達歩数 | トップから2タップ以内に問い合わせ着地 | ファーストビューに問い合わせ導線、または1スクロール以内 |
| ユーザビリティ | コントラスト比 | テキスト4.5:1 以上 | WCAG AA 準拠。黒背景＋ネオンブルー（cyan/sky-400）の可読性確保 |
| ユーザビリティ | 問い合わせフォーム完了時間 | 30秒以内に送信可能 | フィールド数を5項目以下に抑え、入力負担を最小化 |
| 信頼性 | サイト稼働率 | 99.5%以上 | Vercel Hobby SLA に依存 |
| 信頼性 | コンテンツ復元性 | 過去任意時点のバージョンに復元可 | Git履歴 + Vercel デプロイ履歴で担保 |
| 信頼性 | フォーム送信の到達確認 | 送信失敗時にエラー表示と再試行手段を提供 | 送信失敗で問い合わせが消失するとビジネス機会損失 |
| 信頼性 | Resend API 障害時の挙動 | 送信失敗時にフォールバックで mailto: リンクを案内 | サードパーティ依存に対する2段構え |
| セキュリティ | 問い合わせフォームのスパム対策 | bot 送信を90%以上ブロック | honeypot フィールド + Cloudflare Turnstile（無料）の二重防御 |
| セキュリティ | 個人情報の取り扱い | 取得項目を最小化（氏名・メアド・本文のみ）。第三者提供なし | 個人情報保護法・GDPR 準拠 |
| セキュリティ | Resend APIキー管理 | 環境変数で管理（Vercel Environment Variables）、リポジトリにコミットしない | キー漏洩防止 |
| セキュリティ | HTTPS の常時有効化 | 全ページHTTPS | Vercel 標準で実現 |
| セキュリティ | 第三者JavaScript埋め込み | SNS埋め込みは v0.5 以降の楽曲サービスのみ。解析タグは1種類まで | 不要なトラッカー混入による表示遅延と漏洩リスクを抑える |
| スケーラビリティ | 同時アクセス | 月間PV 10,000 まで Hobby プランで対応 | 初期1年は十分。サーキット出演時に Pro プラン($20/月)へ移行検討 |
| スケーラビリティ | 問い合わせフォーム月間処理数 | 月3,000通まで Resend 無料枠で処理可能 | 通常運用で十分。サーキット出演時の出演依頼急増にも対応 |
| スケーラビリティ | 画像最適化 | next/image による自動WebP配信 | バンド写真のサイズが大きくなりがちな問題に標準対応 |
| 対応環境 | ブラウザ | iOS Safari 16+, Chrome 最新2世代, Edge 最新 | ターゲット層（20〜30代音楽ファン）の主要環境 |
| 対応環境 | 画面サイズ | 320px〜1920px の全幅対応 | スマホ縦持ち〜デスクトップフルHDまで |

## 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|----------|
| フレームワーク | Next.js 14 (App Router) | 静的生成 + Route Handlers でフォームのバックエンドも同居可能。FCP/LCPの非機能要件を満たす |
| 言語 | TypeScript | 型による安全性。ライブ情報・メンバー情報のスキーマ化で更新ミスを防ぐ |
| スタイリング | Tailwind CSS | ユーティリティクラスでデザイン一貫性を担保。`bg-black` `text-cyan-400` `text-sky-400` で青×黒テーマを明示的に実装 |
| 画像最適化 | next/image | WebP 自動配信、レスポンシブ最適化が標準。ヒーロー画像とメンバー写真の重さ問題を吸収 |
| デプロイ | Vercel Hobby (無料) | Next.js 最適化、Git push 自動デプロイ、無料SSL、グローバルCDN。月コスト 0円 |
| ドメイン | 初期: `*.vercel.app` / Phase 4: 独自ドメイン | サーキット出演確定時まで遅延 |
| 問い合わせフォーム送信 | **Resend** | 無料3,000通/月、Next.js Route Handler との統合がスムーズ。APIキー管理は環境変数 |

## リスク分析

### 技術的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| 5/31 公開期限の達成困難（14日間のタイトスケジュール） | 7/11 ライブ集客機会の損失 | Phase 1 のスコープを必須機能のみに絞り込む。Week 1 終了時点（5/23）で進捗確認、遅延時は写真ギャラリー・過去ライブ履歴を即座に Phase 2 へ繰り越し |
| アー写撮影のスケジュール調整失敗 | ヒーロー画像が確保できない | バックアップとして既存ライブ写真10枚から選定可能。撮影は 5/24〜25 を目標、最悪 5/28 まで可 |
| 問い合わせフォームのスパム流入 | フォーム機能の運用負荷増 | honeypot + Cloudflare Turnstile の二重対策。週次でスパム到達数をモニタ |
| Resend API 障害 | フォーム送信ができない | エラー時に mailto: リンクへフォールバック表示。Resend のステータスページを定期確認 |
| Vercel 無料枠の超過（月100GB帯域） | サイトが月末に止まる可能性 | Vercel Analytics で帯域監視。Pro プラン移行の閾値を月間PV 5,000 と定める |
| Next.js / Tailwind のメジャーアップデート | 既存実装の破壊的変更 | 依存パッケージのバージョンを `package.json` で固定 |

### ビジネス的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| 5/31 までに公開できず7/11ライブの集客導線が機能しない | 動員数の伸び悩み | Phase 1 のクリティカルパス管理。Week 1 終了時の中間レビュー必須。最悪 6/14 を最終許容期限とし機能を削っても公開 |
| バンド継続が困難になった場合のサイト存続 | サイトが負債化 | Vercel 無料枠なら金銭的負担ゼロ。独自ドメインを Phase 4 まで遅延することで意思決定を後ろに倒せる |
| 競合バンドのサイトとの差別化失敗 | 印象に残らない | ネオンブルー×黒の鮮烈なトーン、サカナクション系の世界観を写真選定とコピーで体現 |
| ブッキング問い合わせの取りこぼし | 出演機会の損失 | F3.6 + F3.3 の二段構え（フォーム ＋ mailto:）。送信失敗時のエラー表示と再試行手段で取りこぼしを最小化 |
| チケット予約URL未取得による予約フロー不明確 | 来場意欲のあるユーザーが離脱 | F1.2 で「会場へお問い合わせください」を明示し、会場の問い合わせ手段を確実に案内 |

### 運用的リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| えんまさ一括管理の属人化（繁忙・体調不良・移動時の更新停止） | ライブ情報の鮮度低下、過去日付の露出 | F5.4 のフォールバック設計（aberyo or ひろむに Git アクセス権付与）。F1.5 の「次回公演調整中」UI で過去日付露出を防ぐ |
| ライブ情報の更新が止まる | バンドが活動停止と誤認される | 月次のサイトチェック日をえんまさのカレンダーに固定（提案: 毎月1日） |
| 写真・動画の権利トラブル | サイト構造に穴 | F5.1 の1営業日削除対応構造、`CONTENT_POLICY.md` での事前合意で予防 |
| 問い合わせフォーム経由の個人情報漏洩 | 信頼失墜、法的リスク | 取得項目を最小化、HTTPS常時化、Resend の認証情報を環境変数管理。プライバシーポリシー明記 |
| アー写撮影タスクが Week 1 のえんまさ業務を圧迫 | 開発と撮影の両立困難 | カメラマン任せでカット選定だけえんまさが対応、撮影の段取りは aberyo（人脈担当）に協力依頼 |

### UX / プロダクトリスク

| リスク | 影響 | 対策 |
|--------|------|------|
| ボーカル動画が無いことで「楽曲未完成のバンド」と新規訪問者に映る | 期待感の低下、離脱 | ヒーロー写真の選定とキャッチコピーで世界観を強く打ち出す。SNSへの導線をファーストビューに明確に置く |
| モバイル離脱（読み込み遅延／タップミス） | せっかくの流入が無駄に | LCP 2.5秒以内・タップエリア44px以上を厳守 |
| 青×黒の配色が暗すぎて読みづらい | 情報伝達失敗 | コントラスト比4.5:1 以上を全要素で検証 |
| 問い合わせフォームの完了率低下（離脱） | ブッキング機会損失 | フィールド数5項目以下、必須項目の明示、送信後のサンクスメッセージ、エラー時の親切なフィードバック |
| 「会場へお問い合わせください」表記がユーザーに不便と映る | チケット予約の心理的ハードル増 | 文言を「ご予約・お問い合わせは会場（Blue Sheep）まで」と丁寧にし、電話番号 or 会場サイトリンクを明示 |

## 付録A: 権利ポリシー（CONTENT_POLICY.md の素案）

```
■ メンバー写真
  - 本人の Yes が出たもののみ掲載
  - 削除依頼から1営業日以内に削除可能な構造を維持
  - メンバー脱退時の取り扱いを事前合意

■ ライブ写真
  - 撮影者の明示（クレジット）
  - 写っているメンバー全員の Yes が出たもののみ
  - 会場が撮影禁止の場合は掲載不可

■ アー写
  - 撮影カメラマンとの利用範囲合意を事前に取り付ける（Web掲載・SNS転載・印刷物 等）
  - クレジット表記の要否を確認

■ 動画
  - ホームページでの動画埋め込みは現フェーズで実施しない
  - SNSプラットフォーム上での運用に一本化

■ 楽曲埋め込み
  - 公式配信プラットフォーム（Spotify / Apple Music 等）の埋め込みのみ可（v0.5以降）

■ ニュース投稿
  - 投稿前にえんまさのレビュー必須
  - 個人を特定する情報（住所・連絡先）は載せない
```

## 付録B: プライバシーポリシー（privacy.md の素案）

```
■ 取得する情報
  - 問い合わせフォームでの送信者氏名・メールアドレス・問い合わせ内容のみ

■ 利用目的
  - 問い合わせへの返信
  - 出演依頼・取材依頼・コラボ依頼への対応

■ 保持期間
  - 問い合わせ対応完了後1年で削除

■ 第三者提供
  - 一切行わない
  - 送信処理に Resend (https://resend.com/) を利用、データはResendを経由するが第三者への提供は行わない

■ 解析ツール
  - Vercel Analytics（Cookieless、個人特定不可）

■ 削除依頼の手段
  - freoli.official@gmail.com への削除リクエストにより1営業日以内に対応
```
