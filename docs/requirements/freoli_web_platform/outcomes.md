# FREOLI Web Platform — Outcomes

> Vision「心が先に動く、ライブへの最終確認所」を **測定可能な成功基準** に変換した層。
> 数値の (仮定) ラベルは、第 1 回測定機会（2025/7/11 Blue Sheep ライブ）後に実測値で校正する。

---

## 1. North Star Metric — 全ライブの総動員数

このプロダクトの成功を **1 つの指標** で表すなら **「全ライブの総動員数」**（KGI）。

**選定根拠**:

1. Vision V-7 North Star Quote の終着点が「絶対観に行く」=「ライブ参加」であり、サイトの成功を最も説明する 1 指標。
2. 既存 v3 §2「主要 KPI / KGI」で **動員数を KGI** と明記済み。
3. AGENTS.md 設計原則：「決断の最終確認所」── ライブ参加への転換が起こったかどうかが、サイトの存在意義そのもの。

**指標の正確な定義**: その月の **全公演の動員数の合計**（v0.1 期はサイト経由かどうかの完全計測が困難なため、サイトの貢献は推定で運用）。

**v0.5 以降の精緻化候補**: 「サイト訪問 → ライブ参加転換率」（Vercel Analytics + 自己申告アンケート併用）。

## 3. 失敗下限値（方向転換シグナル）

### 動員数の失敗下限

| 節目 | 失敗下限値 | 方向転換のシグナル |
|---|---|---|
| 2025/7/11 Blue Sheep | **動員 15 人未満**（目標 30〜40 人の 50% 未達） | サイトの集客機能が機能していない or SNS → サイト導線が機能していない可能性。F1（ライブ集客機能）の見直し、SNS リンクバーの位置・コピーの再検証。 |
| 2025 年内 v0.5 期 | **平均動員 40 人未満 / 回**（50% 未達が 3 公演連続） | 楽曲リリースを行っても動員に転換しない = サイト以前の課題。本サイトを単独で改修しても効果は薄い前提で、戦略全体の見直しを起動。 |
| 2027 年初頭 v1.0 期 | **100 人未達のまま 1 年経過** | サーキット出演条件未達 = ロードマップ全体の遅延。Vision §2「3 年後の差分」シナリオの見直し検討。 |

### 動員数以外の失敗シグナル

- **問い合わせフォーム 0 件 / 3 ヶ月連続**: ブッキング窓口として機能していない（v3 KPI「月 3 件以上」の下限）。F3.6 フォーム送信フローのバグ調査と、SNS DM への流出度合いの確認。
- **サイト訪問者数 100 unique / 月未満**（ライブ告知期を除く）: SNS → サイト導線が壊れている可能性。SNS リンクバーやプロフィール URL の確認。

## 4. 計測方法

### 4.1 北極星指標「全ライブの総動員数」

| 項目 | 値 |
|---|---|
| データソース | 会場入場カウント（ライブハウス報告 or 自前手集計） |
| 計算式 | `その月の全ライブ動員数の合計` |
| 計測頻度 | **ライブごとに即日記録**、月次でレポート集計 |
| 担当 | えんまさ（Content Owner）が公演翌日に記入 |

**実装提案（D-O1）**: `data/lives.ts` に `actual_attendance: number | null` フィールドを追加し、公演翌日に Content Owner が記入。Git 履歴で時系列の動員数推移を追える。

### 4.2 KPI 群

| KPI | データソース | 計測頻度 | 備考 |
|---|---|---|---|
| **サイト訪問者数（unique visitors）** | Vercel Analytics（無料枠） | 月次 | v0.1 期から計測。ライブ告知期 ±1 ヶ月を主要監視。 |
| **問い合わせフォーム着信数** | Resend メール受信ログ + Gmail ラベル `freoli/inquiry` 自動付与 | 月次 | `freoli.official@gmail.com` 受信箱で月初に手集計。 |
| **SNS フォロー転換率** | サイト訪問数 ÷ SNS フォロー増加数（自己申告アンケート併用） | 月次 | **完全な計測は不可**（SNS のリファラ取得は限界あり）。「概算」と明記して運用。 |
| **Spotify / Apple Music 初回再生数** | 各プラットフォームの artist 管理画面 | 月次 | **v0.5 リリース後**から計測開始。 |

### 4.3 計測コストの段階方針

- **v0.1 期**: Vercel Analytics + Resend ログ + 手集計のみ（GA4 等の本格的解析ツールは導入しない）
- **v0.5 期**: 必要なら GA4 検討（ただし AGENTS.md の Reversibility-First と権利ガバナンス観点から、Cookie 追加にはプライバシーポリシー更新が必要）
- **v1.0 期**: 独自ドメイン取得後に GA4 / 類似ツール本格導入を検討

## 5. セグメント別の観点

**v0.1 期方針**: セグメント分けせず **全体値のみ** で運用（サンプル数が少なく統計的有意性が確保できないため）。

### v0.5 期以降に検討する切り口

| 切り口 | 何が分かる | 計測手段 |
|---|---|---|
| **ファン vs 業界**（問い合わせフォーム送信時の選択肢） | ブッキング窓口としての機能度合い | フォームに「お問い合わせ種別」プルダウン追加（D-O2） |
| **SNS チャネル別流入**（リファラ取得） | どの SNS が最も決断後押しに効いているか | Vercel Analytics の Referrer 情報（精度は限界あり） |
| **モバイル vs デスクトップ** | ペルソナ高橋みさきの想定流入経路（Instagram = モバイル中心）と実態のギャップ | Vercel Analytics の Device 情報 |
| **ライブ告知期 vs 平常期** | サイトの「ライブ集客装置」としての効果度合い | 月次レポートで前後比較 |

**v0.1 への先取り実装（D-O2）**: F3.6 問い合わせフォームに「お問い合わせ種別」プルダウン（ファン感想 / 出演依頼 / 取材 / その他）を追加。極小コストで業界 vs ファン内訳が即座に取れる。

## 6. Leading / Lagging KPI 体系

### 6.1 Leading KPI（先行指標 = ユーザー行動・週次〜月次）

| KPI | 何を表す | 関連 Vision/Feature |
|---|---|---|
| **サイト訪問者数（unique / 月）** | SNS → サイト導線の通過量 | V-7「気になって HP 開いたら」 |
| **ヒーロー直下までスクロール率**（≒ 直帰の逆数） | ファーストビューの「3 秒で分かる」の機能度 | V-7「一瞬で分かって」/ F2.1 |
| **チケット導線クリック率**（チケット情報セクションのリンク CTR） | 行動転換寸前の意思 | V-7「もう予約まで終わってた」/ F1.2 |
| **SNS リンクバー クリック率** | 流入元以外の SNS への橋渡し効果 | AGENTS.md 非交渉シーケンス「SNS リンクバー」 |
| **問い合わせフォーム表示 → 送信完了率** | フォームの摩擦の少なさ | F3.6 |

### 6.2 Lagging KPI（遅行指標 = ビジネス成果・月次〜四半期）

| KPI | 何を表す | 関連 |
|---|---|---|
| **ライブ動員数 / 公演**（=KGI / 北極星指標） | サイトの究極的成功 | §1, §2 |
| **問い合わせフォーム月間着信数** | ブッキング窓口としての成果 | v3 KPI「月 3 件以上」 |
| **SNS 総フォロワー数の月次増分** | 「あの色のバンド」としての記憶定着 | Vision V-7「あの色のバンド」 |
| **Spotify / Apple Music 月間リスナー数** | 楽曲入口としての成果（v0.5 以降） | v3 §戦略的役割 ② |

### 6.3 監視サイクルと診断ルール

- **Leading**: 月次レビュー（Content Owner = えんまさが月初に Vercel Analytics 確認）
- **Lagging**: 公演ごと（動員数）+ 四半期レビュー（その他）

**診断ルール**:

| パターン | 解釈 |
|---|---|
| Leading 下落 + Lagging 維持 | サイト以外の要因（ライブ自体の集客力、SNS）が動員を支えている = **サイトの貢献が薄れているシグナル** |
| Leading 好調 + Lagging 不振 | サイトは機能しているが、**サイトの先（実ライブ体験 or チケット導線）に問題**がある可能性 |

## 7. 定性的成功シグナル

数値以外で「成功している」とわかるシグナルを 3 Tier で運用。

### Tier 1: 強い成功シグナル（このどれかが発生したら明確に成功軌道）

1. **ライブ会場で「サイト見て来ました」と声をかけられる**: V-7 North Star の現実化。物理的な行動転換が起きた瞬間の証言。
2. **ブッカーから「サイトで動員力を確認した上で出演オファー」が届く**: Secondary Persona 田村さん想定の挙動が現実化。プレスキット機能の成功。
3. **メディア（ライブハウスのブログ、Web 系音楽メディア）から取材依頼**: 業界での「公式サイトが整っている若手バンド」ポジション獲得。
4. **対バン候補バンドから「コラボ・対バン依頼」**: 業界内ネットワークでの認知獲得。

### Tier 2: 中位シグナル（複数積み重なれば成功軌道）

5. **「あの色のバンド」「ネオンブルーの」という外部からの呼称**: V-7「あの色のバンド」の Branding 資産化。X/Twitter 検索などで言及確認。
6. **SNS 投稿で「FREOLI のサイト見て来た」という旨のポスト**: V-7 の発話に近い言及。
7. **ライブ前後で SNS フォロワーが急増**: ライブ → SNS 回帰の好循環が起きている証拠。
8. **問い合わせフォーム経由のコラボ・ブッキングが SNS DM 経由を上回る**: 正規窓口としての機能定着。

### Tier 3: 弱いシグナル（参考程度）

9. **メンバー個人 SNS への「サイトかっこいい」コメント**: 主観的な評価は変動が大きいので参考程度。

### 計測・記録方法（D-O3）

- **シグナル発生簿**は Notion or Google Docs の運用ログとして **外部記録**（リポジトリにコミットしない）。理由: コメント投稿者の個人情報が混ざるため、CONTENT_POLICY 準拠で OSS 化しにくい。
- 月次レビュー時に **Tier 1 が発生したかをカウント**。四半期で 0 件なら定性面でも黄信号。

## 8. 運用品質指標

### 8.1 運用品質

| 指標 | 閾値 | 測定方法 | 関連設計原則 |
|---|---|---|---|
| **ライブ情報の鮮度** | 過去日付のライブを **24 時間以上** トップに残さない | `data/lives.ts` の最新ライブ日付と現在日時の差分を月次確認 | F1.5「ライブ予定の未定状態 UI」 |
| **写真・コンテンツの権利確認率** | 掲載中の写真の **100%** が被写体本人の Yes ログを保持 | Notion/Google Docs の「Yes ログ」と `public/images/` を月次照合 | AGENTS.md 設計原則 ③ / CONTENT_POLICY |
| **削除依頼への対応時間** | **1 営業日以内** に削除完了（PR マージまで） | 削除依頼受領日と該当画像削除コミット日のラグ記録 | AGENTS.md §6 Layer 2 / 設計決定 #15 |
| **Vercel ビルド成功率** | **100%**（main マージ前の preview ビルドが必ず成功） | Vercel ダッシュボードで月次集計 | AGENTS.md §7 機能完了の定義 |
| **問い合わせフォーム送信成功率** | **99% 以上**（Resend 送信成功 ÷ 全送信試行） | Resend ダッシュボード + Vercel Functions ログで月次確認 | F3.6 / 壊れ方 ② 状態不整合 |
| **Turnstile bot 防御の有効性** | スパム着信 **0 件 / 月** | 問い合わせフォーム着信内容の手目検 | F3.6 / 非機能要件 |

### 8.2 信頼性指標（Reversibility-First）

| 指標 | 閾値 | 関連 |
|---|---|---|
| **Vercel revert 復旧時間** | 不具合発生時の 100% で revert が機能（最大 **5 分以内**に前バージョン稼働） | AGENTS.md 設計原則 ② |
| **`.env*` ファイル誤コミット件数** | **0 件**（hook で自動ブロック） | AGENTS.md §6 Layer 1 / `.claude/hooks/prevent-destructive-command.js` |

### 8.3 非機能性能指標（v3 非機能要件由来）

| 指標 | 閾値 | 関連 |
|---|---|---|
| **LCP（Largest Contentful Paint）** | **2.5 秒以内**（モバイル 4G 想定）= V-7「3 秒で分かる」の前提条件 | v3 §非機能要件 |
| **CLS（Cumulative Layout Shift）** | **0.1 以下** | v3 §非機能要件 |
| **可用性（uptime）** | **99% / 月**（Vercel Hobby SLA 準拠の達成度） | v3 §非機能要件 |
