# FREOLI Web Platform 詳細要件定義書

## 1. プロジェクト概要

FREOLI Web Platform は、東京・関東を拠点とする4人組インディーロックバンド「FREOLI」の公式Webサイトである。本サイトの戦略的役割は「新規リスナー発見の入口」ではなく「**決断の最終確認所**」── すなわち SNS や口コミで FREOLI に興味を持った人物が、**ライブ参加 と 楽曲再生** という二大行動への転換を受ける場として設計される。

実現方法としては、Next.js 14 (App Router) + Tailwind CSS + Vercel Hobby + Resend という最小構成のスタックで、リンクツリー型 ＋ ポートフォリオ型のハイブリッドサイトを構築する。楽曲未完成期はバンド写真とライブ告知を主役にネオンブルー × 黒のビジュアルで世界観を打ち出し、問い合わせフォーム（送信先: `freoli.official@gmail.com`）を v0.1 から実装することで、ブッキング窓口としての機能も最初から備える。`v0.1（2025/5/31）→ v0.5（楽曲リリース時）→ v1.0（サーキット出演確定時に独自ドメイン取得）` の3段階リリースで、バンドの節目に同期して進化させる。

## 2. ビジネス要件

* **対象課題**:
  * SNS で FREOLI に興味を持った人物が、ライブ参加・サブスク再生・SNSフォローという行動転換に至る前の「最後の一押し情報」を得られない状況
  * ライブハウス・メディア・他バンドからの問い合わせ窓口が SNS DM に分散し、ブッキング機会の取りこぼしが発生するリスク
  * 楽曲リリース前のため、「このバンドはどんなバンドなのか」を集約して伝える単一の窓口が存在しない

* **提供価値（Value Proposition）**:
  * 興味を持った瞬間に **「次のライブはいつ・どこで・どうやって行けるか」** を3秒で把握できる
  * **ネオンブルー × 黒の鮮烈なビジュアル** で、サカナクション/BUMP系統の音楽性が一目で伝わる
  * **ブッキング・取材・コラボ依頼の正規窓口** を提供し、業界人からの信頼を獲得する

* **主要KPI / KGI**:
  * **動員数（KGI）**: 7/11ライブで30〜40人 → 年内ライブで60〜80人 → 2027年初頭に100人 `(仮定)`
  * **サイト訪問者数**: 7/11 ライブ前1ヶ月で 500 unique visitors `(仮定)`
  * **問い合わせフォーム月間着信数**: 月3件以上（出演依頼・取材・コラボ含む）`(仮定)`
  * **サイト → SNS フォロー転換率**: 訪問者の20%以上が SNS リンクをクリック `(仮定)`

* **競合優位性（7Powers）**:
  * **Cornered Resource（独占資源）**: 祐介（ゆうすけ）のボーカル。業界人（ゆうたさん）からも「歌で何を伝えるかが核」と評価されており、FREOLIにしか出せない声質と表現力は他バンドが模倣不可能な資源。サイトのヒーロー領域・メンバープロフィールでこの非対称性を強調する
  * **Counter-Positioning（カウンターポジショニング）**: 多くのインディーバンドが Lit.Link 系の汎用ページに依存する中で、ネオンブルー × 黒の独自世界観と、ライブ集客に特化した行動転換型UIで差別化。「リンクを並べるだけのHP」ではない設計思想自体が競合との断絶を作る
  * **Branding（ブランディング）**: サカナクションを主要リファレンスとする観察的・色彩的な歌詞美学（userMemories 由来）と、ネオンブルー × 黒のビジュアルトーンが一貫することで、「あの色のバンド」として記憶に残るブランド資産を蓄積する

  * **セカンダリ**: ライブハウスのブッカー、メディア関係者、対バン候補バンド。出演依頼や取材・コラボの可能性を探って FREOLI のプロフィールを確認する業界関係者

## 4. 機能要件詳細

### 4.1. ライブ集客機能（F1）

* **概要**: サイト訪問者を「次のライブへの参加」という具体的な行動に転換させる、最優先カテゴリ。トップページのファーストビュー直下に常設し、いかなる訪問者導線でも避けて通れない位置に配置する
* **対応する Stage 1 要件**: F1.1, F1.2, F1.3, F1.4, F1.5
* **業務フロー（正常系・新規ライブ告知）**:
  1. えんまさが新規ライブの情報を収集（日付・会場・時刻・チケット情報・対バン）
  2. メンバーに Slack 等で確認、Yes を得る
  3. ローカルで `data/lives.ts` ファイルの該当配列に新規ライブ情報を追加
  4. Git に commit、GitHub に push、PR を作成（運用ルール上は1人レビュー必須だが現状はえんまさのセルフマージ可）
  5. Vercel が自動的にビルド・デプロイ、3分以内に本番反映
  6. 該当ライブが「次のライブ」になる場合、F1.1 のファーストビュー表示が自動更新
* **業務フロー（正常系・訪問者がライブ予約に至る流れ）**:
  1. 訪問者が SNS リンクから https://freoli.vercel.app に着地
  2. ファーストビューでバンド写真と「FREOLI」のロゴを認識（1秒）
  3. すぐ下のライブ情報セクションで「2025/7/11 (FRI) @ Blue Sheep」を確認（3秒）
  4. 「ご予約・お問い合わせは会場（Blue Sheep）まで」をタップ
  5. Google Maps 表示 or 会場サイト遷移 or 電話番号タップ
  6. 会場経由で予約完了
* **例外系 / エラーフロー**:
  * **次回ライブが未確定**: F1.5 により「次回公演調整中」のプレースホルダー表示。「決まり次第 SNS でお知らせします」と SNS リンクへ誘導
  * **過去日付のライブが残っている**: F1.5 の表示制御で、日付が今日より過去になったライブは自動的に「過去のライブ」セクションへ移行（実装では日付比較ロジックで対応）
  * **会場URLが取得できていない**: 会場名のテキストと「会場へお問い合わせください」表示のみで完結。ユーザーが Google で会場名検索する想定
* **特殊要件 / 制約**:
  * F1.2 は「URLが取得できれば置換可能」な構造とする（`venueUrl` 等のオプショナルフィールドをデータ構造に持たせ、有無で表示分岐）
  * ファーストビュー直下に必ずライブ情報セクションを置く（ヒーロー → ライブ情報 → メンバー → SNS → フォーム → フッターの順を厳守）
  * **F1.6（2026-05-18 grill 派生）**: `data/lives.ts` の型に `actual_attendance: number | null` を含め、Content Owner（えんまさ）が公演翌日に実測値を記入する運用とする。`null` 許容（未記入は KGI 算出から除外、減点しない）。NextLive と PastLives で共通スキーマを採用（F1.1 / F1.4 / 設計決定 #22）

### 4.2. アーティストプレゼンス機能（F2）

* **概要**: バンドの世界観・メンバー像を視覚的・情報的に伝える。楽曲未完成期において最重要となる「FREOLIってどんなバンド？」への回答装置
* **対応する Stage 1 要件**: F2.1, F2.2, F2.3, F2.4（F2.5 は撤回）
* **ユースケース**: ペルソナ「高橋みさき」が Instagram で見たアーティストの世界観を、ホームページで再確認・補強する。ペルソナ「田村ブッカー」が「このバンドどんな感じ？」を3秒で判断する
* **業務フロー（正常系・ヒーロービジュアル表示）**:
  1. 訪問者がページにアクセス、ヒーロー領域に4人集合写真が全幅で表示
  2. 写真上にネオンブルーのアクセントとバンド名「FREOLI」が浮かぶ
  3. その下に1行キャッチコピー（30〜80字）が静的に表示
  4. スクロールするとライブ情報・メンバープロフィールへ自然遷移
* **業務フロー（正常系・メンバー紹介の表示）**:
  1. メンバー4人分のカードが横並び（モバイルでは縦並び）で表示
  2. 各カードには通称（ゆうすけ／あのむ／ひろむ／Aberyo）・パート・顔写真・1〜2行紹介
  3. 紹介順は楽器順（Vo/Gt → Gt → Ba → Dr）を採用、Aberyo はDrとして最後
* **例外系**:
  * **ヒーロー画像の読み込み失敗**: 黒背景＋バンド名のテキストのみで成立する設計（フォールバックレイアウト）
  * **メンバー写真の一部欠落**: パートを示すシルエットアイコンで代替表示
  * **撮影予定のアー写が間に合わない**: 既存ライブ写真10枚から横長対応の集合写真を選定して暫定使用
* **特殊要件 / 制約**:
  * ヒーロー画像は **next/image** の `priority` 属性を付けて初期ロード最適化
  * 画像は WebP 配信、`sizes` 属性でレスポンシブ最適化
  * 撮影タスクは Week 1 中（5/24〜25 目標）に完了させる必要

### 4.3. 問い合わせフォーム機能（F3.6）

* **概要**: ブッキング・取材・コラボ依頼を一元的に受け取る正規窓口。SNS DM の分散・取りこぼしを防ぐ
* **対応する Stage 1 要件**: F3.6, F5.5
* **ユースケース**: ペルソナ「田村ブッカー」がライブ出演を打診したい時に使用。あるいはメディア関係者が取材依頼で使用
* **業務フロー（正常系・フォーム送信）**:
  1. 訪問者がフォームセクションに到達、4つのフィールド（名前・メアド・**用件カテゴリ（5 種: 出演依頼 / 取材・メディア / コラボ / ファン感想 / その他）**・本文）と Cloudflare Turnstile を表示
  2. 訪問者が必須項目を入力（Turnstile の bot 判定をパス）
  3. 送信ボタンをタップ
  4. クライアント側でバリデーション（メアド形式、本文10文字以上、honeypot フィールドが空であること）
  5. Next.js Route Handler `/api/contact` に POST
  6. サーバー側で Turnstile トークンを Cloudflare API で検証
  7. Resend API 経由で `freoli.official@gmail.com` 宛にメール送信（送信者名・連絡先メアド・本文を含む）
  8. 送信完了レスポンスを返却、クライアントで「送信ありがとうございました」画面表示
  9. えんまさが Gmail でメールを受信、用件カテゴリに応じて返信
* **業務フロー（正常系・問い合わせ対応）**:
  1. えんまさが Gmail でフォーム経由のメールを受信
  2. 用件カテゴリで優先度判定（出演依頼 > 取材 > コラボ > **ファン感想** > その他）
  3. 24時間以内に1次返信を目標
  4. 必要に応じてメンバーに Slack で共有・相談
  5. 1ヶ月以内に問い合わせ対応完了、1年後にメール内容を削除（プライバシーポリシー準拠）
* **例外系 / エラーフロー**:
  * **Turnstile 検証失敗**: 「人間であることを確認できませんでした、もう一度お試しください」と表示、フォーム入力内容は保持
  * **honeypot フィールドに値あり**: bot 判定、送信せず正常終了風レスポンスを返す（攻撃者への情報露出を防ぐ）
  * **Resend API 障害**: エラー画面で「送信に失敗しました、お手数ですが `freoli.official@gmail.com` まで直接ご連絡ください」と mailto: リンクを案内
  * **必須項目未入力**: クライアント側で送信ボタン非活性化、入力フィールド下にエラーメッセージ
  * **メアド形式不正**: クライアント側バリデーションで即時フィードバック
* **特殊要件 / 制約**:
  * Resend API キーは **Vercel Environment Variables で管理**、リポジトリにコミットしない
  * Turnstile サイトキー・シークレットキーも環境変数管理
  * 送信先メアド `freoli.official@gmail.com` の管理アカウント所有はえんまさが把握しておく（アクセス権の引き継ぎ可能な状態）

### 4.4. SNS / サブスク連携機能（F3.1〜F3.5）

* **概要**: 訪問者を FREOLI のSNSアカウントとサブスク楽曲（v0.5以降）へ誘導する補助機能
* **対応する Stage 1 要件**: F3.1, F3.2, F3.3, F3.4, F3.5
* **業務フロー（正常系）**:
  1. 訪問者がページ下部のリンクセクションに到達
  2. Instagram / YouTube / TikTok / X / Spotify / Apple Music / mailto: の7つのボタンが grid 表示
  3. 各ボタンをタップで該当 SNS / サブスクアプリ or 新規タブで開く
  4. mailto: ボタンタップでメーラー起動、`freoli.official@gmail.com` 宛の新規メール画面
* **例外系**:
  * **楽曲リリース前の Spotify / Apple Music ボタン**: grayed out 状態、または「2025年配信予定」テキスト表示。タップしても何も起こらない、もしくは「Coming Soon」モーダル表示
  * **SNS アカウントが削除された場合**: 該当ボタンを非表示にする（リンクデータの管理側で制御）
* **特殊要件**:
  * 外部リンクは必ず `target="_blank"` + `rel="noopener noreferrer"`（F3.5）
  * アイコンは Lucide React など軽量なアイコンライブラリを使用

### 4.5. ニュース機能（F4）

* **概要**: ライブ告知・楽曲リリース・メディア出演を時系列で発信。コアファンとの継続的な接点を作る副次機能
* **対応する Stage 1 要件**: F4.1, F4.2, F4.3
* **業務フロー（正常系・ニュース投稿）**:
  1. えんまさがニュース内容を起案（ライブ告知・楽曲リリース等）
  2. ローカルで `data/news.ts` に新規エントリを追加（日付・タイトル・本文・タグ）
  3. Git に commit、push、自動デプロイ
  4. ニュース欄に最新3〜5件が時系列降順で表示
* **例外系**:
  * **ニュース欄が空（運用初期）**: 「初回投稿準備中」のプレースホルダーを表示しない代わりに、7/11ライブ告知を初回投稿として用意
* **特殊要件**:
  * 月1回以上の更新を運用目標（チェック日: 毎月1日をえんまさのカレンダーに固定）

### 4.6. 運用・ガバナンス機能（F5）

* **概要**: コンテンツの権利管理・更新ロール・削除リクエスト対応・プライバシー保護を仕組みで担保
* **対応する Stage 1 要件**: F5.1, F5.2, F5.3, F5.4, F5.5
* **業務フロー（コンテンツ削除リクエスト対応）**:
  1. メンバー本人または問い合わせ経由で削除依頼を受信
  2. えんまさが該当コンテンツ（写真・テキスト・リンク）を特定
  3. Git で該当データ行を削除する commit を作成
  4. push → Vercel 自動デプロイ → 3分以内に本番反映
  5. 削除完了を本人または依頼者に通知（1営業日以内）
* **業務フロー（担当不在時のフォールバック）**:
  1. えんまさが3週間以上対応不可になる兆候（出張・体調・移動等）が出る
  2. aberyo または ひろむ に GitHub リポジトリの Write 権限を一時付与
  3. 該当者がライブ情報・ニュースの最低限の更新を実施
  4. えんまさ復帰後、権限を元の状態に戻す（または継続付与の判断）
* **特殊要件**:
  * `CONTENT_POLICY.md` と `/privacy` ページは v0.1 公開時点で整備済みであること
  * Vercel・Resend・Cloudflare・GitHub の管理アカウント情報はえんまさが安全に保管、緊急時のアクセス手順を文書化

## 5. 非機能要件

* **UI / UX**:
  * モバイルファースト設計（想定流入経路の90%以上がモバイル）
  * 1スクロール以内に「次のライブ情報」が到達
  * タップ可能エリア最低44×44px
  * ファーストビュー → ライブ → メンバー → SNS → フォーム → フッターの固定スクロール順
  * 黒背景＋ネオンブルーで WCAG AA 準拠（コントラスト比 4.5:1 以上）
  * 問い合わせフォームの完了所要時間 30秒以内、フィールド数5項目以下

* **パフォーマンス**:
  * First Contentful Paint (FCP) ≦ 1.5秒（4G環境）
  * Largest Contentful Paint (LCP) ≦ 2.5秒
  * Cumulative Layout Shift (CLS) ≦ 0.1
  * ライブ情報更新の本番反映 ≦ 3分（Vercel 自動デプロイ標準）

   * **送信先**: `freoli.official@gmail.com`（固定）

2. **Cloudflare Turnstile**:
   * **用途**: 問い合わせフォームのbot対策（CAPTCHA代替）
   * **連携方式**: クライアント側で Turnstile ウィジェット表示、サーバー側で `siteverify` API へ POST
   * **認証方式**: サイトキー（公開）＋ シークレットキー（環境変数 `TURNSTILE_SECRET_KEY`）
   * **想定レート / コスト**: 無料、レート制限実質なし
   * **データ種別**: トークンのみ送受信、個人情報は経由しない

3. **Vercel（ホスティング + Analytics）**:
   * **用途**: 静的・SSR ホスティング、自動デプロイ、グローバルCDN、Cookieless Analytics
   * **連携方式**: GitHub リポジトリ連携による自動デプロイ、Vercel Dashboard で環境変数管理
   * **認証方式**: GitHub OAuth
   * **想定コスト**: Hobby プラン無料、月間PV 10,000 まで対応。Pro プラン移行は月間PV 5,000 到達時に検討（$20/月）

4. **GitHub**:
   * **用途**: バージョン管理、コード・コンテンツの履歴管理、PRレビューによるガバナンス担保
   * **連携方式**: HTTPS / SSH での push/pull
   * **認証方式**: Personal Access Token または SSH Key
   * **アクセス権**: えんまさ（Owner）、aberyo / ひろむ（Write、フォールバック発動時のみ）
   * **想定コスト**: プライベートリポジトリは無料枠で十分

   * **連携方式**: `next/font/google` 経由でセルフホスト相当（ビルド時にフォント取得）
   * **コスト**: 無料

6. **Google Maps（外部リンクのみ）**:

   * **コスト**: 無料

## 7. 技術選定とアーキテクチャ

* **フレームワーク**: Next.js 14 (App Router)
* **言語**: TypeScript 5+
* **フロントエンド**: React 18 + Tailwind CSS 3+
* **バックエンド**: Next.js Route Handlers（API Routes）。問い合わせフォームの送信処理に限定
* **状態管理**: ローカル State（`useState`）のみ。グローバル状態管理は不要（静的サイトに近いため）
* **データ永続化**:
  * コンテンツデータ: TypeScript ファイル直書き（`data/lives.ts`, `data/members.ts`, `data/news.ts`）
  * 永続的なDBは持たない（問い合わせフォームの送信内容も Resend 経由で送信のみ、サーバー側に保存しない）
* **認証**: なし（v0.1〜v1.0 で一貫してサイト側認証は不要）
* **AI連携**: なし
* **ホスティング / デプロイ**: Vercel Hobby（Edge Network、自動 SSL）
* **外部メール送信**: Resend
* **bot 対策**: Cloudflare Turnstile + サーバー側 honeypot 検証

* **アーキテクチャ方針**:
  * **静的生成優先（SSG）**: ライブ情報・メンバー情報・ニュースはすべてビルド時に静的生成、訪問者から見ると単純な静的サイトとして配信される（最速ロード）
  * **動的処理は API Route のみ**: 問い合わせフォーム送信のみ Route Handler でサーバー処理。それ以外はすべて静的配信
  * **データ更新フロー**: Git push → Vercel ビルド → 静的ファイル再生成 → CDN 配信（コンテンツ更新は実質 3 分以内）
  * **コンテンツとコードの分離**: `/data/` ディレクトリ配下にコンテンツデータをファイル別に整理（lives.ts, members.ts, news.ts）。将来的な CMS 移行時にこのディレクトリのみ置換可能な構造
  * **環境変数管理**: 秘匿情報（API キー）は Vercel Environment Variables、公開可能な設定（サイトURL等）は `next.config.js` で管理
  * **権利ポリシー駆動**: `CONTENT_POLICY.md` をリポジトリの README 直下に置き、コミット前の確認を促す運用

## 9. リスクと課題

* **5/31 公開期限の達成困難（14日間のタイトスケジュール）**: 7/11 ライブ集客機会の損失
  * *対策*: Week 1 終了時点（5/23 金曜日）に必ず中間レビューを実施。アー写撮影調整・基盤構築・データ構造のいずれかで遅延が発生していれば、即座に **写真ギャラリー（F2.4）と過去ライブ履歴（F1.4）を Phase 2 へ繰り越し**判断を行う。最悪のフォールバックとして 6/14 を最終許容期限とし、機能を更に削ってでも公開する

* **アー写撮影のスケジュール調整失敗**: ヒーロー画像が確保できない
  * *対策*: 5/24〜25 撮影目標。aberyo（人脈担当）が知り合いカメラマンとの調整を主導。撮影できない場合は、既存ライブ写真10枚から横長対応の集合写真を選定して暫定使用（5/28 までに代替画像確定）

* **問い合わせフォームのスパム流入**: フォーム機能の運用負荷増、Resend無料枠の浪費
  * *対策*: honeypot フィールド（hidden な `<input name="website">`）+ Cloudflare Turnstile の二重対策。週次でスパム到達数を Resend ダッシュボードでモニタ、増加時は Turnstile のセキュリティレベルを上げる（managed → invisible へ）

* **Resend API 障害**: 問い合わせフォームの送信が機能しない
  * *対策*: エラー時に `freoli.official@gmail.com` への mailto: リンクをフォールバック表示。Resend の Status ページ（https://status.resend.com/）を定期確認。送信失敗のログを Vercel Functions のログで保全し、後追い対応可能にする

* **チケット予約URL未取得**: 来場意欲のあるユーザーが予約方法に迷う
  * *対策*: F1.2 で「ご予約・お問い合わせは会場（Blue Sheep）まで」を明示。会場の電話番号 or 公式サイトリンクを明示的に案内（中優先度の追加質問で会場連絡先を確定する）

* **えんまさ一括管理の属人化**: 繁忙・体調不良・移動時の更新停止
  * *対策*: F5.4 のフォールバック設計を運用化。GitHub のリポジトリ設定で aberyo or ひろむ に Read 権限を恒久付与、緊急時のみ Write 権限を一時付与する手順を Notion に文書化。月次サイトチェック日（毎月1日）をえんまさのカレンダーに固定リマインダー設定

* **ボーカル動画が無いことで「楽曲未完成のバンド」と新規訪問者に映る**: 期待感の低下、離脱
  * *対策*: ヒーロー写真の選定とキャッチコピーで世界観を強く打ち出す（サカナクション系の観察的・色彩的表現）。SNS への導線をファーストビューに明確に配置し、「動画はSNSで見てね」という自然な誘導を作る

* **アー写撮影タスクが Week 1 のえんまさ業務を圧迫**: 開発と撮影の両立困難
  * *対策*: 撮影段取りは aberyo に協力依頼、えんまさは撮影当日のディレクション・写真選定のみ担当。開発・撮影が同日に重ならないようカレンダー調整
