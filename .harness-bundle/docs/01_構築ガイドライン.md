# ハーネス 01 — 構築ガイドライン

> 新規リポジトリに対して、AIエージェント（Claude Code等）が事故なく作業できる「ガードレール」を構築するための入門ドキュメント。
> 「リスク判定 → 保護モデル選定 → 初期 PR ガバナンス確立」までの一本道。

---

## 0. なぜハーネスが必要か

AIエージェントは速度に優れるが、**「壊れたときの影響範囲」と「修復可能性」を判断できない**。
人間が判断するための時間も毎回取れない。よって：

- **インフラ層**（DB schema / 認証 / 環境変数）は壊れると修復不能 → **コマンドレベルで物理ブロック**
- **コンテンツ層**（ブランド・契約に関わるもの）は壊れると信頼を失う → **PR レビュー必須**
- **開発層**（UI・ロジック）は壊れても Git で revert 可能 → **AI 委任 OK**

これが本ガイドラインの基底にある **3層保護モデル**。

---

## 1. Step 0 — リスク判定

最初にプロジェクトのリスクレベルを判定する。判定根拠を `docs/harness/HARNESS_DECISIONS.md` に書き残すこと。

### 判定マトリクス

| リスクレベル | 該当条件 | ハーネスの強さ |
|---|---|---|
| **R1** | 個人ブログ・OSS・本番ユーザーなし | hook 不要、PR 規約のみ |
| **R2** | ユーザーDB なし / 認証なし / 本番影響あり | 軽量 hook + PR レビュー |
| **R2+** | DB あり または 認証あり、課金なし、ユーザー数 〜1000 | 本ガイドラインの推奨セット |
| **R3** | 課金あり または 1000人超 または PII保管あり | R2+ に加えて CI gating・本番DB分離 |

### R2+ への昇格トリガー

R1/R2 で始めたプロジェクトでも、以下のいずれかが発生したら R2+ に書き換える：

- ユーザーDB（Supabase / Firestore / 自前 DB）の導入
- 認証（Clerk / Auth0 / 自前 JWT）の導入
- 第三者 API キー（OpenAI / Stripe テストモード等）の利用開始
- SNS 告知や招待リンクで本番ユーザーが入ってくる

### R3 への昇格トリガー

- 課金機能（Stripe 本番 / Apple IAP 等）
- ユーザー数 1000 人超
- PII（フルネーム・住所・電話番号）の保存
- SLA コミットを伴う B2B 契約

## 2. Step 1 — 3層保護モデルの設計

リスク判定が済んだら、プロジェクト固有の保護対象を 3 層に分類する。

### Layer 1：インフラ層（hook で自動ブロック・人間が手動でのみ実行）

**判定基準：壊れると Git revert で戻せない、または全ユーザー影響**

| 典型例 | 保護手段 |
|---|---|
| DB マイグレーション破壊コマンド | hook で実コマンドブロック |
| `.env*` ファイル読み取り | hook で `cat` 等をブロック |
| 認証ミドルウェア設定 | CODEOWNERS で人間承認必須 |
| `git push --force` / `git reset --hard` | hook でブロック |
| 危険な SQL（`DROP TABLE` / RLS 無効化 等） | hook で grep ブロック |

→ **`hooks/prevent-destructive-command.js`** が信頼源（同梱）。

### Layer 2：コンテンツ層（変更前に意味設計担当の承認必須）

**判定基準：壊れるとブランド・契約・信頼が失われる、修復に時間を要する**

| 典型例（プロジェクト依存） | 保護手段 |
|---|---|
| ブランドコピー・公開文言 | PR レビュー必須（CODEOWNERS） |
| AI プロンプトテンプレート | PR レビュー必須 |
| 契約・利用規約に関わるロジック | PR レビュー必須 |
| 価格・課金ルール | PR レビュー必須 |

→ **`AGENTS.md` に「Protected Areas」セクションを設けて明示**。hook で全パターン捕捉は不可能、人間の判断に委ねる。

### Layer 3：開発層（AI 委任 OK）

**判定基準：壊れても Git で revert 可能、影響範囲が局所**

| 典型例 | 保護手段 |
|---|---|
| UI コンポーネント | PR レビュー任意 |
| API ルート（型契約 OK な範囲） | Gate 1 のみ |
| 単体テスト | レビュー任意 |
| 開発スクリプト | レビュー任意 |

## 3. Step 2 — Minimum Fence, Maximum Speed

### 原則

「壊れたら困るもの **だけ** をフェンスする。それ以外は通す。」

過剰なフェンスは AI を止めるだけでなく、人間の開発速度も殺す。MVP 段階では特に重要。

### 通すべき操作（ブロックしない）

- `rm -rf node_modules` — 依存の再インストールは通常作業
- `pnpm install` / `pnpm add` / `npm install` — パッケージ管理は通常作業
- `git push`（force なし） — feature ブランチへの通常 push
- `next dev` / `next build` / `vite build` 等 — ビルドは可逆
- `cat <ソースファイル>` — コード読み取りは漏洩リスク低い
- `gh` / `curl` / `jq` — 外部 API コール（インフラ操作ではない）

### ブロックすべき操作（同梱 hook の対象）

| カテゴリ | 例 |
|---|---|
| DB 破壊系 | `supabase db reset` / `supabase db push` / `supabase migration repair` |
| SQL 破壊系 | `DROP TABLE` / `TRUNCATE` / RLS 無効化 / 全権限ポリシー |
| 秘密情報読み取り | `cat .env*` / `cat supabase/.env` |
| Git 破壊系 | `git push --force` / `git reset --hard` |
| 危険な rm | `rm -rf` 引数なし |

→ プロジェクト固有のブロック対象は **`hooks/prevent-destructive-command.js` の `DESTRUCTIVE_PATTERNS` 配列を編集** して追加する。配列が信頼源。

## 4. Step 3 — レビューゲートの設計（Gate 1 / Gate 2）

AI が生成した全ての PR は以下の 2 層ゲートを通過してからマージする。

### Gate 1 — 技術的適合性（全 PR 必須）

技術設計担当（バックエンド or フルスタックエンジニア）がチェック：

| チェック項目 | 観点 |
|---|---|
| 認可（IDOR 防御等） | ユーザースコープが全 CRUD で確認されているか |
| 認証連携 | JWT / セッション取得が共通ヘルパー経由か |
| 型安全境界 | API 入力値が Zod 等で検証されているか |
| N+1 / 過剰 select | ループ内 DB 呼び出しなし / 不要カラム露出なし |
| エラー境界 | 内部エラーオブジェクトをそのままクライアントに返していないか |

### Gate 2 — 意味的妥当性（意味設計を含む変更のみ）

意味設計担当（プロダクトオーナー or デザインリーダー）がチェック：

| 変更タイプ | 添付すべきもの |
|---|---|
| AI プロンプト変更 | AI 応答サンプル 3 件（before/after） |
| UI コピー・文言変更 | プレビュー URL + 変更文言 diff |
| UX フロー・画面遷移変更 | プレビュー URL + フロー図 |
| ブランド要素変更 | プレビュー画像 |

**Gate 2 対象外**（意味設計担当の承認不要）:
- スタイリングのみ変更
- バグ修正（表示内容変更なし）
- パフォーマンス改善・リファクタリング

### 機械の床 / 人間の門

```
機械が自動実行する床:  typecheck + lint + build （静的解析）
人間が下す門:           Gate 1 / Gate 2 のチェックリスト （仕様適合）
```

「ビルドが通ったから完了」は禁止。完了の定義は **型チェック通過 + ビルド成功 + 動作確認済み（人間）** の 3 条件。

## 5. Step 4 — 初期 PR ガバナンス（Bootstrapping Paradox の解消）

### 問題

初期セットアップ PR は以下の構造的問題に陥りやすい：

```
CODEOWNERS:   @ProjectOwner  ← Owner だけ
  ↓
Owner が PR 作成者 → 自分の PR を自分で承認できない（GitHub ルール）
  ↓
他メンバーはまだコラボレーター未招待 → 誰もマージできない
```

これは「審判が自分の試合を裁けない」問題。**Bootstrapping Paradox** と呼ぶ。

### 採用する 3 原則

**① Transparent Bypass（透明なバイパス）**
Admin bypass は「こっそりルールを外す」のではなく、**Ruleset Bypass List に明示的に設定**する。GitHub の Audit Log で追跡可能にし、「緊急例外」を「設計上の正当な経路」に変える。

**② Honest CODEOWNERS（現実に正直な所有者定義）**
招待されていないメンバーを CODEOWNERS に記載しない。チームの現状を反映した設計を保ち、コラボレーター招待後に全員記載に更新する。

**③ Bootstrap PR ≠ Feature PR（初期ガバナンス PR の別扱い）**
ルール自体を作る PR と、ルールに従う PR は本質的に異なる。初期段階は Admin 権限者が Bootstrap PR を bypass マージし、以降の全 PR から正規フローを適用する。

### 進め方

| フェーズ | アクション |
|---|---|
| Bootstrap 期 | Ruleset Bypass List に Admin（Owner）を追加 |
| Bootstrap 期 | 初期 PR を Admin bypass でマージ |
| コラボレーター招待後 | 他メンバーを Collaborators に招待 |
| コラボレーター招待後 | CODEOWNERS を全員記載に更新 |
| 初の正規フロー | 更新 PR を他メンバーにレビューしてもらう |

### 避けるべき選択肢

| 選択肢 | 却下理由 |
|---|---|
| メンバー招待 → 承認待ち | 招待受諾まで時間がかかる。即マージ不可 |
| Code Owners required を一時オフ | 手動戻し忘れリスク。透明性ゼロ |
| 直接 main push | force push 扱いで Ruleset がブロック |

## 6. Step 5 — アトミック確認ループ

### 完了の定義

> **機能完了 = 型チェック通過（機械）＋ ビルド成功（機械）＋ 動作確認済み（人間）**

3 条件が揃うまで main マージ不可。

### 壊れ方 3 パターン（タスク着手時に該当を宣言する）

| パターン | 症状 | 確認手段 |
|---|---|---|
| ① ロジック破綻 | 同一入力で異なる結果 | 単体テスト |
| ② 状態不整合 | 書き込んだのに読めない（認可 + 認証） | プレビュー URL + 手動ログ確認 |
| ③ UX 断絶 | API は動くが UI が仕様と違う | プレビュー URL での目視確認 |

タスク着手時に「このタスクは ①〜③ のどれを起こしうるか」を宣言し、対応する確認手段を計画する。

### PR チェックリスト雛形

`.github/pull_request_template.md` に以下を入れる：

```markdown
## Gate 1（技術的適合性）— 全 PR 必須
- [ ] 認可・認証連携・型安全・N+1・エラー境界 の 5 点確認済み
- [ ] レスポンス構造が型定義と一致

## Gate 2（意味的妥当性）— 意味的変更を含む場合
- [ ] AI プロンプト / ロジック変更 → 応答サンプル 3 件添付済み
- [ ] UI コピー / UX フロー変更 → プレビュー URL + 文言 diff 添付済み
- [ ] Gate 2 対象外（スタイリング・バグ修正のみ）→ その旨を明記

## 共通
- [ ] typecheck 通過
- [ ] build 通過（またはプレビューデプロイ成功）
- [ ] 動作確認済み（パターン①〜③ の該当を確認）
```

## 7. Step 6 — ブランチ管理規則

### 命名規則

| プレフィックス | 用途 |
|---|---|
| `feature/<slug>` | 新機能開発 |
| `fix/<slug>` | バグ修正 |
| `docs/<slug>` | ドキュメント |
| `chore/<slug>` | 設定・依存関係・CI |
| `refactor/<slug>` | リファクタリング |

- `<slug>` は kebab-case（例：`feature/42-user-profile`）
- 日本語・スペース禁止
- Issue 番号プレフィックスは任意

### 3 原則

- **Merge Once, Delete Immediately**：PR マージ後、ブランチを自動削除（GitHub Settings の「Automatically delete head branches」ON）
- **Name First, Automate Second**：命名規則は宣言のみで、違反は PR コメントで警告（マージブロックなし）
- **Long-term Branches are Explicit**：長期育成ブランチは `.github/PROTECTED_BRANCHES.txt` に明示列挙

## 8. Step 7 — ファイル配置と初期セットアップ

新規リポジトリへの導入手順：

### ディレクトリ構成

```
<repo>/
├── .claude/
│   ├── settings.json        ← hook を有効化
│   ├── hooks/
│   │   └── prevent-destructive-command.js   ← 同梱
│   └── skills/              ← .skill を展開して配置
├── .github/
│   ├── CODEOWNERS           ← 意味設計担当・技術設計担当を記載
│   ├── pull_request_template.md  ← 同梱テンプレート参照
│   └── PROTECTED_BRANCHES.txt    ← 長期ブランチ列挙（あれば）
├── AGENTS.md                ← Protected Areas を明記（同梱テンプレート参照）
├── CLAUDE.md                ← `@AGENTS.md` 1 行のみ可
└── docs/
    └── harness/
        ├── HARNESS_DECISIONS.md   ← リスク判定根拠を書く
        └── HARNESS_HEALTH.md      ← 月次レビュー記録
```

### 構築手順

1. **このバンドルを展開** — `dist/band-platform-harness/` を新リポジトリ直下に展開
2. **`install.sh` 実行** — `.claude/skills/` に skill 群を配置
3. **`AGENTS.md` のテンプレート** を `templates/AGENTS.md.template` からコピーし、プロジェクト固有の値で穴埋め
4. **`.claude/settings.json`** をマージ（hook を有効化）
5. **CODEOWNERS** を作成 — 意味設計担当・技術設計担当・Layer 2 パスを記載
6. **GitHub Ruleset 設定**
   - main ブランチ保護（PR 必須・承認 1 件以上）
   - Bypass List に Admin（Owner）を追加（Bootstrapping 期のみ）
7. **PR テンプレート配置** — `.github/pull_request_template.md` を `templates/pull_request_template.md` からコピー
8. **`HARNESS_DECISIONS.md`** を `docs/harness/` に作成 — Step 0 のリスク判定根拠を記入

## 9. 月次レビュー（HARNESS_HEALTH）

ハーネスは固定資産ではなく、プロジェクトと共に進化させる。月次で以下を確認：

| 確認項目 | アクション |
|---|---|
| ブロックされた誤検知 | hook の `DESTRUCTIVE_PATTERNS` を調整 |
| 通った危険操作 | 新パターンを追加 |
| Gate 2 漏れ | Protected Areas を AGENTS.md に追記 |
| リスクレベル変化 | HARNESS_DECISIONS.md を更新 |

記録は `docs/harness/HARNESS_HEALTH.md` に追記する。

## 付録 A：参考にした設計判断

本ガイドラインは COCOSiL V2 プロジェクトのハーネス設計（R2+ 相当）を汎用化したもの。
原典の議論ログは以下に対応：

- 3 層保護モデル → 「壊れたときの影響範囲」による分類
- Minimum Fence, Maximum Speed → 過剰なフェンスは開発を止める
- Bootstrapping Paradox → 「審判が自分の試合を裁けない」問題
- アトミック確認ループ → 機械の床 / 人間の門

各原則の詳細な議論経緯は本ガイドラインの範囲外（必要なら原典リポジトリの `docs/discussions/` を参照）。
