# Skill Improver — スキル自己改善ループ

既存スキルの「期待と実際のギャップ」を特定し、SKILL.md本文とAssetsを対話的に改善するスキル。

## 基本思想

スキルは使うほど「ここがちょっと違う」が見えてくる。
その気づきを放置せず、都度スキル自体に反映していくことで、使うたびに精度が上がる自己改善ループを作る。

改善で大事なのは **「なぜ期待と違ったか」の根本原因を突き止めること**。
表面的な修正（"この一文を足す"）ではなく、根本原因に対処する変更を提案する。

---

## 起動フロー

このスキルには2つの入口がある。ユーザーの発言からどちらかを判断して進む。

### モードA: 直後フィードバック
ユーザーがスキルを使った直後に「ここが違う」とフィードバックしている場合。
直前の会話コンテキストからスキル名・実際の出力・期待とのズレを読み取れることが多い。

### モードB: 振り返り改善
ユーザーが過去の使用体験を振り返って改善したい場合。
「〇〇スキルを改善したい」のような発言が典型。

---

## ステップ1: 対象スキルの特定と読み込み

1. ユーザーの発言から対象スキルを特定する
2. 特定できない場合は `/mnt/skills/user/` を一覧表示し、どれか選んでもらう
3. 対象スキルの SKILL.md を全文読み込む
4. Assets（templates/, references/, fewshots/, examples/ など）があれば構成も把握する

```bash
# スキル一覧の確認
ls /mnt/skills/user/

# 対象スキルの構成確認
find /mnt/skills/user/<skill-name>/ -type f

# SKILL.md読み込み
cat /mnt/skills/user/<skill-name>/SKILL.md
```

スキルが読み取り専用の場合は、作業コピーを作成する:
```bash
cp -r /mnt/skills/user/<skill-name> /home/claude/<skill-name>-wip
```

---

## ステップ2: ギャップの収集

ユーザーからギャップ情報を引き出す。押し付けがましくならないよう、まずは自由に話してもらい、足りない部分だけ補足で聞く。

### モードAの場合（直後フィードバック）
直前の会話からわかる情報を先にまとめ、ユーザーに確認する:

> **私の理解**: 〇〇スキルを使って△△を作成したところ、□□の部分が期待と違った
> - 期待していたこと: ...
> - 実際の出力: ...
> - ズレのポイント: ...
>
> この認識で合っていますか？補足があれば教えてください。

### モードBの場合（振り返り改善）
以下を対話で把握する（一度に全部聞かず、1〜2問ずつ）:

- **どんな場面で使ったか**: どういうタスクに対してスキルを使ったか
- **何が期待と違ったか**: 具体的にどの部分がズレていたか
- **どうなっていれば良かったか**: 理想の出力・振る舞いのイメージ

過去の会話を検索できる場合は `conversation_search` を使って具体例を探す。

### 根本原因の特定に効く補足質問

ユーザーの最初の説明だけでは根本原因が絞り込めないことが多い。
以下の質問は、原因の切り分けに特に有効:

- **「どの段階でその値/指示を伝えましたか？」** — ヒアリング時の入力ミスなのか、途中の変換で失われたのかを区別できる
- **「毎回起きますか？特定の条件のときだけですか？」** — 再現性で根本原因の種類が変わる（常時＝指示不足、時々＝文脈理解不足）
- **「期待通りだった部分はどこですか？」** — 壊れていない部分を知ることで、問題の範囲を絞り込める

ただし一度に聞きすぎないこと。ユーザーの説明で十分な場合は補足質問なしでステップ3に進んでよい。

---

## ステップ3: 根本原因の分析

収集したギャップ情報をもとに、根本原因を分類する。
references/gap_taxonomy.md を参照し、以下の分類に当てはめる。

### 原因カテゴリ（よくある順）

| カテゴリ | 典型的な症状 | 改善対象 |
|---------|------------|---------|
| **指示不足** | やるべきことが書かれておらず、スキルが自己判断で外す | SKILL.md |
| **指示過剰** | 厳密すぎて柔軟性がなく、少し違う入力で破綻する | SKILL.md |
| **文脈理解不足** | ユーザーの意図や背景を汲み取れていない | SKILL.md |
| **例・テンプレ不足** | 出力の形式やトーンがブレる | Assets |
| **参照情報不足** | 必要な知識・定義・仕様が与えられていない | Assets |
| **手順の順序問題** | 手順の依存関係が考慮されず、前後関係で破綻する | SKILL.md |
| **スコープ曖昧** | 何をやるべきで何をやらないかが不明確 | SKILL.md |

分析結果をユーザーに共有する:

> **分析結果**: このギャップの根本原因は「**指示不足**」と考えます。
> 具体的には、〇〇の場面でどう振る舞うかがSKILL.mdに記載されておらず、
> スキルが汎用的な対応をしてしまったことが原因です。
>
> 合っていますか？別の見方があれば教えてください。

## ステップ4: 改善案の提案

根本原因に対応する具体的な変更を提案する。

### 提案のフォーマット

改善案は必ず以下の形式で提示する:

```
## 改善提案

**根本原因**: [カテゴリ] — [1行の説明]

### 変更1: [変更の概要]
- **対象ファイル**: SKILL.md（or Assets内の具体ファイル名）
- **変更箇所**: [セクション名 or 行の近辺]
- **なぜこの変更が必要か**: [根本原因との対応]
- **変更内容**:
  [具体的な追加・修正テキスト]

### 変更2: ...（必要なら複数）
```

### 提案時の注意

- **「なぜ」を必ず説明する**: 変更の理由が明確でないと、ユーザーが判断できない
- **汎化を意識する**: 目の前の1ケースだけでなく、類似ケース全般に効く変更にする。特定の入力にしか効かないピンポイント修正は避ける
- **最小限の変更を心がける**: 一度に大きく書き換えるのではなく、根本原因に対応する最小の変更を提案する
- **MUSTの乱用を避ける**: 「なぜそうすべきか」を説明すれば、モデルは理解して従える。ALL CAPSの強制より理由の説明が効く

---

## ステップ5: 確認と適用

### 既存ファイルの変更 → 差分の提示
変更を適用する前に、具体的な差分をユーザーに見せる。

対象ファイルの該当箇所について:
- **変更前**（現在の記述）
- **変更後**（提案する記述）

を並べて提示し、確認を求める。

### 新規ファイルの作成 → 内容の提示
Assets（リファレンス、テンプレート等）を新規作成する場合は、差分ではなく以下を提示する:
- **ファイルパス**: どこに何という名前で作るか
- **内容のプレビュー**: ファイルの全文（短い場合）または構成と主要部分
- **正確性の注意**: 技術的なリファレンス（規格値、仕様表など）を新規作成する場合、
  Claudeの知識に基づいて作成するため値が不正確な可能性がある。
  「この値は私の知識ベースに基づいています。正式な規格文書で確認いただけると安心です」と伝える。

ユーザーの確認を得てから作成する。

### 適用
ユーザーの承認を得たら、直接ファイルを書き換える。

```bash
# 読み取り専用の場合は作業コピーに対して編集
# str_replace ツールで該当箇所を書き換える
```

書き換え後、変更が正しく反映されたことを確認する:
```bash
# 変更箇所の前後を表示して確認
```

### 複数変更がある場合
変更は1つずつ適用し、都度確認を取る。
まとめて適用してよいかユーザーに聞いてもOKだが、デフォルトは1つずつ。

---

## ステップ6: 改善ログの記録

改善内容を対象スキルの `changelog.md` に追記する（なければ新規作成）。

```markdown
## YYYY-MM-DD: [改善の概要]
- **きっかけ**: [直後フィードバック or 振り返り]
- **ギャップ**: [何が期待と違ったか — 1行]
- **根本原因**: [カテゴリ — 1行の説明]
- **変更内容**: [何をどう変えたか — 箇条書き]
```

このログがあることで:
- 同じ問題の再発に気づきやすくなる
- 改善の方向性が一貫しているか振り返れる
- 将来の大きなリファクタリング判断の材料になる

---

## ステップ7: 次のアクションの提案

改善適用後、以下のいずれかを提案する:

1. **すぐ試す**: 改善したスキルを使って、先ほどと同じ（or 類似の）タスクを実行してみる
2. **他の改善点はないか確認**: 今回のフィードバックに関連する別のギャップがないか聞く
3. **完了**: 今回はここまで

## 補足: 環境ごとの注意

### Claude.ai
- スキルファイルは `/mnt/skills/user/` にあり、読み取り専用
- 編集は `/home/claude/` にコピーしてから行い、最終版を `/mnt/user-data/outputs/` に出力する
- ユーザーは出力されたファイルをダウンロードし、自分でスキルを差し替える

### Claude Code
- スキルファイルを直接編集できる場合は、そのまま書き換える
- `git diff` で変更内容を確認できるので活用する

---

## やってはいけないこと

- ユーザーの確認なしにファイルを書き換えること
- 「とりあえずMUST/NEVERを足す」という表面的な修正
- 目の前の1ケースにしか効かない過学習的な変更
- 一度に大量の変更を提案して判断負荷を上げること
- 根本原因の分析をスキップして修正に飛ぶこと
