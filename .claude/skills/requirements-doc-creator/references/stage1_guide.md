# Stage 1 詳細ガイド：システム要件定義書の生成

このファイルは `SKILL.md` の Step 4（Stage 1 の生成）の前に必ず読むこと。各セクションの書き方・避けるべきパターン・Few-shot 例を記載。

---

## セクション別の書き方

### システム概要（300〜500文字）

**目的**: 読み手（プロダクトの全体像を初めて見る人）が **30秒で理解できる** 概要を提供する。

**書く順序**:
1. **何を解決するか**（課題）→ 2. **どう解決するか**（ソリューション概要）→ 3. **誰に提供するか**（ターゲット）→ 4. **何で実現するか**（主要技術要素）

**避けるパターン**:
- ❌ 機能の羅列だけ（What ばかりで Why がない）
- ❌ 抽象的すぎる表現（「最先端のAIで」「革新的な」など中身がない）
- ❌ 500文字を大きく超える（読まれない）

**良い例（Task-Hack より）**:
> 「Task-Hack」は、ADHD（注意欠如・多動症）当事者向けに特化したデスクトップ型タスク管理アプリケーションです。既存の Todo 管理ツールが抱える「完了タスクが蓄積することによる認知過負荷」や「計画立案時の麻痺（Activation Gap）」といった課題を解決するため、航空管制（ATC）のUIメタファーと、最新のAI技術（GPT-4o）による「擬似ボディ・ダブリング」および「タスクの自動構造化」を融合させました。

→ **誰の何の問題を、何で解決するか**が30秒で分かる。

### 情報の前提と仮定

**目的**: この要件定義書がどんな素材に基づいているかを明示し、読み手が「ここまでは確定、ここからは仮定」を識別できるようにする。

**書き方**:
- **情報源**: 参照したファイル名・ヒアリング・既存ドキュメントを箇条書き
- **置いた仮定**: 仮定した項目とその根拠を必ず書く

**例**:
```markdown
### 情報源
- `lean_canvas_v2.md`（リーンキャンバス）
- `persona_misaki.md`（ターゲットペルソナ）
- ユーザーヒアリング（2026-04-30 セッション）

### 置いた仮定
- **初期ユーザー規模**: 1,000人を想定（根拠：MVP段階での平均的なスタートアップ規模、リーンキャンバスのチャネル仕様から推定）
- **想定ARPU**: ¥3,000/月（根拠：競合の価格帯と過去の議論ログから）
```

### 確定済み設計決定（Design Decisions Log）

**目的**: 対話を経て確定した設計判断を **資産として残す**。後で「なぜこの仕様にしたか」を辿れるようにする。

**書くタイミング**:
- ユーザーが対話で「Aではなくこっちで」「これは要らない」「これは必須」など判断を下した瞬間に必ず追記
- 仮定を立てた項目で、ユーザーが追認したものも記録

**フォーマット厳守**:
| # | 質問 | 回答・確定内容 | 反映先 |
|---|------|--------------|--------|

「反映先」には影響を受ける機能ID（F2.4 など）またはセクション名を書く。

---

### 機能要件

**目的**: 開発チームが実装に着手できるレベルで、機能を ID 付きで列挙する。

**構造**:
```
## 機能要件

### F1: {コア機能カテゴリ1}
| ID | 要件 | 優先度 | 詳細 |

### F2: {コア機能カテゴリ2}
| ID | 要件 | 優先度 | 詳細 |
```

**カテゴリ分けの原則**:
- 1カテゴリに **3〜7個** の要件が入る粒度で分割
- カテゴリ名は機能領域（「タスク管理」「AIチャット」「データ保存」）。技術用語ではなくユーザー視点で
- F1, F2 の番号は **コアから順** に並べる（F1 が最も重要）

**優先度の使い分け**:
- **必須**: これがないとプロダクトとして成立しない（MVP に含まれる）
- **重要**: なくても動くが、提供価値が大きく毀損される
- **任意**: あれば嬉しい、なくても代替手段で凌げる

**詳細欄の書き方**:
- **動詞ではじめる**: 「タスクを4ゾーンで管理する」ではなく「ACTIVE / NEXT ACTION / HOLDING / CLEARED の4ゾーンを1画面に表示」
- **数値を含める**: 「上限を設ける」ではなく「上限チェック付き（ACTIVE 1, NEXT ACTION 5）」
- **判断材料になる詳細を含める**: 後続の Stage 2 で「業務フロー」を書くための情報を埋め込む

**避けるパターン**:
- ❌ 「ユーザーフレンドリーなUI」のような測定不可能な表現
- ❌ 1つの要件に複数機能を詰め込む（「タスクの作成・編集・削除・並べ替え」→ 分けるべき）

### 非機能要件

**目的**: 「動けばいい」では足りない品質要件を、**測定可能な目標値** で定義する。

**カテゴリ**（網羅すべき）:
- パフォーマンス（応答時間、スループット、起動時間）
- ユーザビリティ（操作の予測可能性、画面遷移、アクセシビリティ）
- 信頼性（データ損失、障害時のフォールバック）
- セキュリティ（認証、データ保護、APIキー管理）
- スケーラビリティ（同時接続数、データ量）
- 対応環境（OS、ブラウザ、デバイス）

**目標値の書き方**:
- **数値で書く**: 「速い」ではなく「100ms以内」
- **計測方法を書く**: 「D&D操作のレスポンスタイム」のように何を計るか明示
- **根拠を必ず書く**: 「なぜ100msか」を「ヒューマンインタラクションの限界値」のように説明

**良い例（Task-Hack より）**:
| カテゴリ | 要件 | 目標値 | 根拠 |
|---------|------|--------|------|
| パフォーマンス | アプリ起動時間 | 3秒以内 | 起動が遅いとADHDユーザーは別のことを始めてしまう |
| パフォーマンス | タスク操作のレスポンス | 100ms以内 | D&D・ボタン操作は即座に反映されるべき |

→ 目標値が数値で、根拠が **プロダクトの本質** に紐づいている。

### 技術スタック

**目的**: 採用する技術の **選定理由** を明示する。「これが流行っているから」ではなく「機能・非機能要件を満たすから」を書く。

**書き方の原則**:
- レイヤーごとに分ける（基盤 / フロントエンド / バックエンド / データ / 認証 / AI / ビルド）
- 各技術の選定理由は **機能・非機能要件のどれに紐づくか** を明示

**良い例**:
| レイヤー | 技術 | 選定理由 |
|---------|------|----------|
| デスクトップ基盤 | Electron | ローカルファイルシステムへの直接アクセス（JSON, MD読み書き）が必須。Webブラウザでは不可能 |
| AI連携 | OpenAI API (GPT-4o) | マルチモーダル対応（画像解析+テキスト生成）。高品質な自然言語処理 |

→ 「なぜこの技術か」が要件と紐づいている。

---

### 開発ロードマップ

**目的**: 「何から作るか」をフェーズで分け、**各フェーズの目的** を明示する。

**Phase 分けの原則**:
- **Phase 1 = MVP Core**: 最もリスクの高い部分・コアバリューを先に検証
- **Phase 2 以降**: 段階的に機能を追加。前フェーズの依存を考慮した順序

**各フェーズに書くこと**:
- **目的**: なぜこのタイミングでこれを作るか
- **タスクリスト**: 具体的な実装項目

**避けるパターン**:
- ❌ 全機能を Phase 1 に詰め込む
- ❌ Phase ごとの目的が「機能を追加する」になっている（具体性ゼロ）

---

### リスク分析

**目的**: 「想定外」を減らすため、**カテゴリ別** にリスクを洗い出す。

**カテゴリ**:
- **技術的リスク**: ライブラリの限界、パフォーマンス、セキュリティの穴
- **ビジネス的リスク**: コスト増、競合、PMF未達
- **運用的リスク**: データ消失、障害時の対応
- **UX / プロダクトリスク**: ユーザーの想定外の使い方、メタファーの違和感
- **法的 / コンプライアンスリスク**: 個人情報、著作権、規制

**書き方**:
- リスク → 影響 → 対策の3列で書く
- 対策は **具体的に**（「気を付ける」ではなく「アプリ起動時にパス検証を実施」）

---

### 追加情報リクエスト

**目的**: Stage 1 の段階で **未確定の項目** を明示し、Stage 2 までに解消する優先順位を示す。

**書き方**:
- 🔴 高優先度（要件の根幹）/ 🟡 中優先度（仕様の精度）/ 🟢 低優先度（あれば書ける）の3段階
- 質問は「Yes/No or 選択式」で答えられる形が望ましい（ユーザーが答えやすい）

---

## Stage 1 全体のチェックリスト

書き終わる前に必ず確認：

- [ ] システム概要が300〜500文字に収まっている
- [ ] 「情報の前提と仮定」に **すべての情報源** が列挙されている
- [ ] 「確定済み設計決定」が **対話で出たすべての判断** をカバーしている
- [ ] 機能要件は **F1, F2... のカテゴリ構造** になっている
- [ ] 各機能の優先度が **必須 / 重要 / 任意** で明示されている
- [ ] 非機能要件の目標値が **すべて数値または明確な状態** で書かれている
- [ ] 技術スタックの選定理由が **要件と紐づいて** いる
- [ ] 開発ロードマップが **Phase別** になっており、各フェーズに目的がある
- [ ] リスク分析が **4〜5カテゴリ** に分けられている
- [ ] 仮定を置いた項目はすべて **(仮定)** とマーキングされている
- [ ] 未確定項目は **(要確認)** とマーキングされている
- [ ] ファイル名は `{プロジェクト名}_system_requirements.md` 形式

すべて ✅ になったら、Stage 1 完了をユーザーに報告し、レビューゲートへ。
