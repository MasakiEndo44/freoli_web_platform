# 対話パターン集：不足情報を引き出す質問テンプレート

このファイルは `SKILL.md` の Step 3（不足情報の対話的補完）で使う質問パターン集。

**最重要原則**：
- **1ターン1論点**: 1回の質問で複数のことを聞かない
- **Closed → Open**: 選択式で大枠を決めてから自由記述で深掘り
- **仮定先行**: 答えづらい質問は仮定を提示して「これでOKですか？」に変換

---

## 不足情報のカテゴリ別質問パターン

### カテゴリ1：システムの目的・ターゲット（🔴 必須）

#### Q-A. ターゲットユーザーが不明な場合

**Closed（選択式）**:
> 「このプロダクトのプライマリターゲットは、以下のどれに近いですか？」
> 1. 個人ユーザー（B2C）
> 2. ビジネスユーザー（B2B / SaaS）
> 3. 開発者向け（B2D）
> 4. 社内ツール（自社利用）

**Open（深掘り）**:
> 「もう少し具体的に、典型的なユーザー像（年齢層・職業・抱えている課題）を教えてください。1人の架空のユーザー像で大丈夫です。」

---

#### Q-B. システムの目的が曖昧な場合

**仮定先行**:
> 「素材から、このシステムの目的は『{推測した目的}』だと読み取りました。これで合っていますか？違う場合は修正してください。」

→ 仮定を提示することで、ユーザーは「ゼロから書く」のではなく「修正する」だけで済む。

---

### カテゴリ2：機能要件の優先度（🔴 必須）

#### Q-C. 機能リストはあるが優先度が不明

**Closed（選択式・複数選択可）**:
> 「以下の機能の中で、**MVPに絶対必要なもの** はどれですか？（複数選択可）」
> - 機能A
> - 機能B
> - 機能C
> - 機能D

その後：
> 「選ばれなかった機能は『Phase 2 以降』に回しますが、その中で **必須でないが重要** なもの（つまり、なくてもMVPは成立するが価値が大きく毀損するもの）はありますか？」

---

#### Q-D. 機能の境界が曖昧な場合

**仮定先行**:
> 「『AIがタスクを分解する』機能について、私は以下のように解釈しました：
> - ユーザーが自然言語で投げた依頼を、3-5個のサブタスクに分解
> - 分解結果は **ユーザー承認後** にボードに追加（自動追加しない）
>
> この理解で合っていますか？違うパターンがあれば教えてください。」

---

### カテゴリ3：非機能要件の目標値（🟡 推奨）

#### Q-E. パフォーマンス目標が不明

**仮定先行 + 数値提示**:
> 「パフォーマンス要件として、以下を仮定で置きました。それぞれOKか確認してください：
> - 主要画面の初回表示: **3秒以内**（一般的なWebアプリの標準）
> - 操作のレスポンス: **100ms以内**（ヒューマンインタラクションの限界値）
> - API応答: **2秒以内**
>
> 業務特性で異なる値が必要なものはありますか？」

→ 数値を提示することで、ユーザーが「思いつかない」状態を回避。

---

#### Q-F. 想定ユーザー規模が不明

**Closed（選択式）**:
> 「初期のターゲットユーザー規模はどれくらいですか？」
> 1. 100人未満（クローズドベータ・社内利用）
> 2. 100〜1,000人（パブリックベータ）
> 3. 1,000〜10,000人（パブリック展開・初期）
> 4. 10,000人以上（スケールフェーズ）

→ この回答で「スケーラビリティ」「インフラ費用」「データ可用性」のすべてが推定可能になる。

---

### カテゴリ4：技術スタック（🟡 推奨）

#### Q-G. 開発チームのスキルが不明

**Closed**:
> 「フロントエンド開発は誰が担当しますか？」
> 1. ユーザー自身が単独で実装
> 2. 開発パートナー / 外注
> 3. 社内エンジニアチーム

その後：
> 「使い慣れている / 採用したい技術スタックはありますか？（例：React / Vue / Swift など）特になければ、要件から推奨スタックを提案します。」

---

#### Q-H. 外部API依存の意思決定

**仮定先行**:
> 「AI機能の実装方法について、以下を仮定しました：
> - **OpenAI API（GPT-4o）** を使用
> - APIキーはユーザーが自前で提供（BYOK = Bring Your Own Key）
>
> これで進めてOKですか？それとも別のモデル（Anthropic Claude、Gemini など）を希望しますか？」

---

### カテゴリ5：ビジネス要件（🟡 推奨）

#### Q-I. KPI / KGI が不明

**Closed**:
> 「このプロダクトの **最重要KPI** は以下のどれに近いですか？」
> 1. ユーザー数（DAU / MAU）
> 2. 継続率（リテンション）
> 3. 収益（MRR / ARPU）
> 4. エンゲージメント（セッション時間 / アクション数）

その後（Open）:
> 「具体的な目標値はありますか？（例：3ヶ月後に MAU 1,000 / 6ヶ月後に MRR ¥100,000 など）。仮定でも構いません。」

---

#### Q-J. 競合優位性 / 差別化が不明

**Open + 構造化された問い**:
> 「既存の類似サービスと比べて、このプロダクトの **独自性** はどこにありますか？以下のどれかに当てはまるものがあれば教えてください：
>
> - **アプローチが違う**（既存とは逆の方針）
> - **特定のユーザー層に特化**（既存は汎用、自社はニッチ）
> - **データ / コンテンツの優位性**（独自に蓄積している資産）
> - **コスト構造の違い**（既存より圧倒的に安い / 高品質）
> - **その他**: {自由記述}」

→ 7Powers のフレームワークに自然に誘導する質問。

---

### カテゴリ6：開発スケジュール（🟢 任意）

#### Q-K. 開発期間が不明

**仮定先行**:
> 「開発スケジュールは仮定で『MVP まで 8〜12 週間 / フル機能 6 ヶ月』と置きました。実際の希望期間があれば教えてください。」

→ 期間が確定していない場合は仮定で進めて (仮定) と明記すれば十分。

---

### カテゴリ7：運用・コスト（🟢 任意）

#### Q-L. ランニング費用の制約

**Closed**:
> 「ランニング費用（月額）の上限イメージはありますか？」
> 1. ゼロに近い（個人開発・趣味レベル）
> 2. 月数千円まで（個人プロ運用）
> 3. 月1〜5万円（小規模スタートアップ）
> 4. 月5万円以上（本格運用）
> 5. 制約なし（成長重視）

## 対話の流れの典型例

### パターン1：素材が薄いケース

```
Claude: 確定要件を整理しました。以下が現時点で読み取れる内容です：
- システム名: COCOSiL
- カテゴリ: AIチャット型自己理解アプリ
- ターゲット: (要確認)
- 主な機能: (要確認)
- 技術スタック: (要確認)

Claude: まず最も重要な3点をお聞きします。

Claude: 【Q1】このプロダクトのプライマリターゲットは、以下のどれに近いですか？
1. 個人ユーザー（B2C）
2. ビジネスユーザー（B2B）
3. 社内ツール

User: 1（個人ユーザー）

Claude: 了解しました。次に【Q2】を聞かせてください。
```

→ **絶対に Q1 と Q2 を一度に聞かない**。1ターン1論点。

---

### パターン2：素材が豊富なケース

```
Claude: 素材から以下の要件を読み取りました：
- システム名: COCOSiL
- ターゲット: 自己理解を深めたい個人（20代〜30代女性中心）
- コア機能: AIチャット / MBTI診断 / 占星術 / 動物占い / 六星占術
- 技術: Next.js / Supabase / Clerk / OpenAI Agent Builder

(仮定) を置いた項目：
- 想定ユーザー規模: 1,000人（MVP初期）
- 月額課金: ¥1,000〜¥5,000

これで Stage 1 を生成してOKでしょうか？それとも、特定のセクション（例：非機能要件、KPI設定）について追加で確認したい点はありますか？
```

→ 素材が豊富な場合は、**仮定を提示して『これでOK?』** で済ませる。質問を増やしすぎない。

---

### パターン3：差分更新モード

```
Claude: 既存の要件定義書（v1.0）を読み込みました。以下の差分が想定されますが、合っていますか？

【追加】
- 新機能 X（AI推薦機能）

【変更】
- 機能 Y のスコープを縮小（Phase 2 → Phase 3 に後ろ倒し）

【削除】
- なし

これで進めてOKですか？他にもアップデートしたい点があれば教えてください。
```

→ 差分更新は **既存との対比** で示すことが重要。

## やってはいけない質問パターン

### ❌ 複数質問の同時投下

```
ターゲットユーザーは誰で、想定規模はどれくらいで、収益モデルは何で、
技術スタックの希望はありますか？
```

→ ユーザーが疲弊し、答えが浅くなる。**1ターン1論点を厳守**。

---

### ❌ オープンクエスチョンから始める

```
このプロダクトについて教えてください。
```

→ ユーザーが「何を答えていいか分からない」状態になる。**Closed から始める**。

---

### ❌ 答えにくい比較質問

```
非機能要件の目標値はどうしますか？
```

→ 専門知識がない人には答えられない。**仮定を先に提示** して「これでOK？」に変換。

---

### ❌ 仮定を立てずに「分かりません」で止まる

```
ターゲットユーザーが素材から読み取れないため、Stage 1 の生成を中断します。
```

→ ❌ 絶対にNG。仮定を立てて (仮定) 明記で進めるか、**1問だけ聞いて再開**する。

---

## 追加情報リクエスト（Stage 1 末尾）の書き方

Stage 1 を生成した時点で、まだ未確定の項目を **優先度別** にリスト化する。

**例**:

```markdown
## 追加情報リクエスト

### 🔴 高優先度
1. KPI/KGIの具体的な目標値（現時点では仮定値で記載）
2. 想定する初期ユーザー規模（インフラ要件に直結）

### 🟡 中優先度
1. 開発期間の制約（現時点では Phase 別の目安のみ）
2. ランニング費用の上限（現時点では一般的な仮定値）

### 🟢 低優先度
1. ブランドガイドライン（UIデザインフェーズで必要）
2. 多言語対応の優先度
```

→ ユーザーは Stage 2 に進む前に、**🔴 だけは答えたほうがよい** と分かる。

---

## まとめ：質問するときの心構え

このスキルが目指すのは「**ユーザーに考えさせる時間を最小化** しながら、必要な情報を全て集める」こと。

- **1ターン1論点** を絶対に守る
- **Closed → Open** の順で聞く
- **仮定先行** で「ゼロから書く」を回避
- 仮定で進められる項目は **(仮定) で進めて、後で確認** に回す
- **質問の総量を抑える**（多すぎると離脱する）

良い対話は「答えるたびに自分の考えが整理される」感覚を生む。それを目指す。
