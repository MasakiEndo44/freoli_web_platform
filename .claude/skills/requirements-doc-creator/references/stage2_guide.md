# Stage 2 詳細ガイド：詳細要件定義書の生成

このファイルは `SKILL.md` の Step 6（Stage 2 の生成）の前に必ず読むこと。Stage 1 → Stage 2 のマッピングルール、加筆すべき内容、避けるべきパターンを記載。

---

## Stage 1 → Stage 2 マッピングの原則

Stage 2 は Stage 1 の **拡張版** であり、**置き換え版ではない**。Stage 1 の内容を引き継ぎつつ、以下の3種類の操作を行う：

1. **転記**: Stage 1 の内容をそのまま Stage 2 の対応セクションに移す
2. **再編成**: Stage 1 の複数セクションを Stage 2 の1セクションに統合する
3. **加筆**: Stage 1 にない情報を素材から拾うか、対話で確認して追加する

---

## セクション別マッピング

### 1. プロジェクト概要 ← Stage 1「システム概要」を整形

**操作**: 転記 + 軽い拡張

Stage 1 の「システム概要」をベースに、300〜500文字の概要を書く。Stage 1 で書いたものをほぼそのまま使ってよい。表現の重複が気になる場合は、Stage 2 では **「実現方法」を少し詳細化** する。

**避けるパターン**:
- ❌ Stage 1 と完全に同じ文章をコピペ（読み手は「2つ読む価値がない」と感じる）
- ❌ 全く別の概要を書く（Stage 1 と Stage 2 で印象が変わると混乱する）

---

### 2. ビジネス要件 ← 素材（リーンキャンバス + 7Powers）から構築

**操作**: 加筆（Stage 1 には明示的にない）

**書く項目**:
- **対象課題**: リーンキャンバスの「課題」セクションから
- **提供価値**: リーンキャンバスの「独自の価値提案 (UVP)」から
- **主要KPI / KGI**: 素材になければ仮定で書く（要確認マーク）
- **競合優位性**: 7Powers 分析から、該当する Power を1〜3個ピックアップ

**7Powers の参考**:
- Scale Economies（規模の経済）
- Network Economies（ネットワーク効果）
- Counter-Positioning（カウンターポジショニング）
- Switching Costs（スイッチングコスト）
- Branding（ブランディング）
- Cornered Resource（独占資源）
- Process Power（プロセス優位）

各 Power を「自社にどう当てはまるか」で1-2文で記述。

**良い例（Task-Hack より）**:
> * **Cornered Resource**: ユーザー自身のパーソナルデータ（Obsidian Vault）に直接書き込み、最適化され続けるAIプロファイルの蓄積によるスイッチングコスト。
> * **Counter-Positioning**: 記録や効率を重視する既存のGTDアプリに対し、「タスクを画面から消すこと」「AIが片付けること」を主軸にしたアンチパラダイム。

→ 自社のユニークさが具体的なメカニズムで説明されている。

---

### 3. ユーザー要件 ← 素材（ペルソナ）+ Stage 1（ロードマップ Phase 1-2）

**操作**: 加筆 + Stage 1 の MVP 部分を引用

**書く項目**:
- **ターゲットユーザー**: ペルソナファイルから引用
- **ペルソナ**: 名前・属性・行動パターン・課題・期待を構造化（**架空の名前+具体的な人物像** が望ましい）
- **MVPの定義**: Stage 1 のロードマップ Phase 1〜2 をベースに、**ユーザーから見た MVP の体験** を記述

**MVP定義の書き方**:
- 機能のリストではなく、「ユーザーが MVP を使って何ができるか」を1-2文で記述
- 例: 「ATCダッシュボード（4ゾーン）でタスクを D&D 管理し、ローカル保存とポモドーロタイマーで集中作業ができる状態」

---

### 4. 機能要件詳細 ← Stage 1「機能要件」を業務フロー付きで詳細化

**操作**: 拡張（最も重要なセクション）

**Stage 1 → Stage 2 の進化**:
- Stage 1 では「ID / 要件 / 優先度 / 詳細」の表形式
- Stage 2 では各機能カテゴリを **個別のサブセクション** に展開し、以下を加筆：
  - **概要**: その機能で何を実現するか
  - **対応する Stage 1 要件**: F1.1, F1.2 のような ID 参照
  - **業務フロー（正常系）**: 番号付きステップ
  - **例外系**: エラー時・想定外の状況での挙動
  - **特殊要件 / 制約**: 設計上の重要な前提

**良い例（Task-Hack より）**:
```markdown
### 4.3. AI Co-planner (チャット・タスク自動分解)
* **概要**: 画面下部のチャットUIに自然言語または画像を投下することで、AIがタスクを生成・提案する。
* **業務フロー**:
  1. ユーザー入力：「来週の定例会議の資料作らなきゃ」
  2. AI応答：「承知しました。構成案作成、データ収集、スライド作成の3つに分解してセットしましょうか？」
  3. ユーザー承認後、NEXT ACTIONに3つのタスクカード（フライトストリップ）が生成される。
* **例外系**: オフライン時、またはAPIエラー時はチャット送信ボタンを無効化し、ローカル手動追加のみとする。
```

→ **具体的な対話例** までが業務フローに含まれており、開発者が UI を設計するのに十分な情報がある。

**書くべき機能カテゴリの数**:
- Stage 1 のすべての主要機能カテゴリ（F1, F2, F3...）を 4.1, 4.2, 4.3... として展開
- すべて書くと冗長になる場合は、**コア機能（必須優先度）に絞って詳細化** し、それ以外は Stage 1 の表形式で省略してよい

---

### 5. 非機能要件 ← Stage 1 の同セクション

**操作**: ほぼ転記 + 軽い再構成

Stage 1 の表形式から、Stage 2 では **カテゴリごとの箇条書き** に再構成すると読みやすい。情報量はほぼ同じだが、「UI/UX」「パフォーマンス」「データ可用性とセキュリティ」のようにグルーピングする。

---

### 6. インテグレーション要件 ← Stage 1 の技術スタックから派生

**操作**: 加筆（Stage 1 には独立セクションがない）

**書く内容**:
- 各外部サービスについて：用途・連携方式・認証方式・想定レート/コスト
- ローカル環境連携（ファイルシステム I/O など）も含める

**書き方のコツ**:
- API 仕様の詳細が未確定なら **(要検討)** と明記
- データ連携の方向（読み取り / 書き込み / 双方向）を明示

**良い例**:
```markdown
1. **OpenAI API**:
   * **用途**: 自然言語テキスト解析、画像（Vision）解析によるタスク生成、ユーザープロファイルへの要約。
   * **連携方式**: Electron Mainプロセスから `openai` Node SDK または REST 経由で呼び出し。
```

---

### 7. 技術選定とアーキテクチャ ← Stage 1「技術スタック」+ アーキテクチャ方針

**操作**: 転記 + アーキテクチャ方針の加筆

Stage 1 の技術スタックを引き継ぎつつ、Stage 2 では **アーキテクチャ方針** を加える。例：
- 「メインプロセスとレンダラープロセスを IPC で完全分離」
- 「フロントエンドとバックエンドを REST API で接続、認証は JWT」
- 「マイクロサービス構成、サービス間は gRPC」

**書くべき判断**:
- どのプロセスが何を担当するか
- データの流れ（クライアント → サーバー → DB の経路）
- 認証 / 認可の方針

---

### 8. 開発プロセス / スケジュール ← Stage 1「開発ロードマップ」

**操作**: 転記 + 期間目安の加筆（あれば）

Stage 1 のロードマップを引き継ぐ。Stage 2 では各 Phase に **期間目安**（X週間 / Yヶ月）を加えるとよい。期間が確定していなければ書かなくてよい（仮定で書く場合は (仮定) 明記）。

**開発体制**: 担当者・役割の分担があれば記載。1人開発なら「単独開発」と明記してよい。

---

### 9. リスクと課題 ← Stage 1「リスク分析」

**操作**: 転記 + 対策の具体化

Stage 1 のリスクを引き継ぐが、Stage 2 では **対策をより具体的に書く**。「対策を講じる」ではなく「アプリ起動時にパス検証を実施し、見つからない場合は設定画面に誘導」のようなアクションレベルで。

---

### 10. ランニング費用と運用方針 ← 新規（技術スタックから推定）

**操作**: 加筆

**書く項目**:
- **インフラ費用**: クラウド利用料、または「ローカル運用のためゼロ」
- **API費用**: OpenAI、Stripe、その他の従量課金サービス
- **SaaS費用**: Supabase、Clerk、Vercel など月額固定費
- **運用体制**: 誰が運用するか
- **モニタリング**: ログ・メトリクス収集の方針

**数値の扱い**:
- 確定値が分からなくても、**料金表から仮定で算出** してよい（その場合 (仮定) 明記）
- 例: 「OpenAI API: 月額 $1〜$5（個人利用想定、GPT-4o の標準レート）」

---

### 11. 変更管理 ← 新規

**操作**: 加筆

| 日付 | バージョン | 概要 | 承認者 |
|---|---|---|---|

**新規作成モードの場合**:
- v1.0 の行のみを記載

**差分更新モードの場合**:
- 既存の変更履歴を引き継ぐ
- 新しい行（v1.1, v2.0 など）を追加し、変更内容を記載

---

### 12. 参考資料 ← 新規

**操作**: 加筆

参照したすべての素材ファイル名を列挙：
- Stage 1 のシステム要件定義書ファイル名
- 素材（リーンキャンバス、ペルソナ、競合分析、インタビューログ）
- 外部資料（参考にした記事、論文、公式ドキュメント）

## Stage 2 全体のチェックリスト

書き終わる前に必ず確認：

- [ ] 12章すべてが揃っている
- [ ] 各章が Stage 1 の対応セクションを **無視していない**
- [ ] 機能要件詳細（第4章）に **業務フロー（正常系・例外系）** が書かれている
- [ ] ビジネス要件（第2章）に **7Powers の該当 Power** が1個以上書かれている
- [ ] ユーザー要件（第3章）に **ペルソナの具体像** が書かれている
- [ ] インテグレーション要件（第6章）に **すべての外部連携** が網羅されている
- [ ] ランニング費用（第10章）に **数値またはレンジ** が書かれている（仮定OK）
- [ ] 変更管理（第11章）に **v1.0 の行** が記載されている
- [ ] 参考資料（第12章）に **Stage 1 のファイル名** が含まれている
- [ ] 仮定 / 未確定項目は (仮定) (要確認) でマーキングされている
- [ ] ファイル名は `{プロジェクト名}_detailed_requirements_specification.md` 形式

---

## 差分更新モードの特殊ルール

差分更新モードで Stage 2 を生成する場合：

1. **既存の Stage 2** を `view` で読む
2. 変更点を **既存セクションに反映**（全文書き換えではなく差分のみ）
3. **第11章「変更管理」に新しい行を追加**
4. **第12章「参考資料」に旧バージョンへの参照** を追加
5. ファイル名は `{プロジェクト名}_detailed_requirements_specification_v2.md` のようにバージョン明示

差分の追加項目だけを書くのではなく、**ドキュメント全体としての整合性** を保つこと。
