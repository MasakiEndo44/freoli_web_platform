# QUESTIONS_F.md — Feature 層 定型問い

## この層の目的

「どんな体験として切り出すか」を Feature 単位で定義する。  
Capability に対する「今の実装ベット」。変わってよい層。  
各 Feature は 1 ファイル（`FEAT-NNN_<slug>.md`）。

## 完了条件（Exit Criteria）

1 Feature につき全 8 問が回答済みで、ユーザーストーリーが 1 文で書けること。

## このあと進む層

Feature 完了 → Evals（E）でその Feature の受け入れ基準を書く

## 書き出し先

`docs/requirements/<product>/features/FEAT-NNN_<slug>.md`  
テンプレート: `docs/requirements/_template/feature_template.md`

---

## 定型問い一覧

### F-1: Persona（誰が使う）

**問い**: この feature を主に使うのはどんなユーザー（ペルソナ）ですか？

**推奨回答の探し方**: V-5 の Primary Persona 回答を引用。別ペルソナなら新規定義

**escape hatch**: 「誰でも」→ 「最初に使う 1 人を想像してください」

---

### F-2: トリガー（When）

**問い**: ユーザーがこの feature を使うのはどんな瞬間・状況ですか？（トリガー）

**推奨フォーマット**: 「〔ペルソナ〕が〔状況〕にあるとき」  
例: 「マネージャーが重要な会議の前日」「担当者が報告書を提出した後」

**escape hatch**: 抽象的すぎる場合 → 「具体的に何をしているときですか？パソコンで？スマホで？」

---

### F-3: 完了状態（Done When）

**問い**: この feature を使い終わったとき、ユーザーはどんな状態になっていますか？

**推奨フォーマット**: 「〔ペルソナ〕が〔できた・わかった・終わった〕状態」

**escape hatch**: 「よくなった」「満足した」→ 「具体的に何が変わりましたか？数字や状態で言えますか？」

---

### F-4: 上位 Capability との紐付け

**問い**: この feature はどの Capability（C-N）の上に立ちますか？

**推奨回答**: C 層で定義した capability 一覧を表示して選んでもらう  
「CAP-001: 〇〇できる ← これが対象ですか？」

**escape hatch**: 複数にまたがる場合 → 「最も関係が深い 1 つはどれですか？」

---

### F-5: ユーザーストーリー

**問い**: この feature の 1 文ユーザーストーリーを作りましょう。  
「As a [persona], I want [goal], so that [benefit].」の形式で。

**推奨回答の生成方法**: F-1（persona）+ F-2（goal: トリガーから）+ F-3（benefit: 完了状態から）を組み合わせる

**escape hatch**: 英語が難しい場合 → 日本語でも可:「〔ペルソナ〕として、〔目的〕のために〔行動〕したい」

---

### F-6: 受け入れ基準の核（3 つ）

**問い**: この feature が「完成した」と言えるための受け入れ基準を 3 つ挙げてください。  
（詳細な EARS 記法は E 層で書きますが、今は要点だけ）

**推奨フォーマット**: 「〔条件〕のとき、〔動作〕が確認できること」

**escape hatch**: 3 つ出ない場合 → 「最低限これができれば使えると言えるものは何ですか？」

---

### F-7: Out of Scope（境界）

**問い**: この feature が**含まない**もの・やらないことは何ですか？

**推奨回答例**: 「A/B テスト機能は含まない」「通知機能は Phase 2 以降」

**Web Search Trigger**: 競合プロダクトの feature 範囲を確認することで Out of Scope の妥当性を検証できる
- 発動タイミング: ユーザーが「競合と比べてどうか」「他のツールでは？」と言った時（積極的には発動しない）
- クエリ例: `"<競合プロダクト名> features list 2026"` / `"<カテゴリ> SaaS standard features 2026"`
- ツール: `WebSearch` → 競合の公式サイト・機能ページを確認
- スキップ条件: ユーザーが既に Out of Scope を明示している

**escape hatch**: 「特にない」→ 「F-4 の Capability より広い範囲に触れていませんか？」と確認

---

### F-8: 実装フェーズ

**問い**: この feature はどの実装フェーズに入れますか？（Phase 1 / 2 / 3 等）

**推奨回答**: Phase がまだ未定義の場合 → 「Phase 1（MVP）/ Phase 2（拡張）/ Phase 3（最適化）」の 3 択で仮置き

**escape hatch**: 「すべて Phase 1」→ 「MVP で最低限必要なものだけに絞るとしたら？」

---

## Escape Hatch 共通ルール（F 層）

- Capability の話に戻りかける → 「Capability は確認済みです。今はその体験の切り出し方を話しましょう」
- Engineering の実装詳細に入る → 「それは S 層の話です。今はユーザー体験として何が必要かを話しましょう」
- Feature が大きすぎる → 「これは 1 画面で完結しますか？複数の feature に分けられますか？」
