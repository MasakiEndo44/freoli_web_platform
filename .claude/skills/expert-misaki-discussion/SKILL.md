# Expert × Misaki Discussion Skill

ユーザーの要求に対し、**動的に選定した専門家2人 ＋ 固定キャラ「みさき」（ギャル）** による5ターン議論を行い、議論ログを資産として残しつつ、議論を踏まえた成果物を生成するスキル。

## ⚡ Step 0：実行モードの判定（他のステップに進む前に必ず完了させる）

スキル読み込み直後、**Step 1 より前に**以下の判定を完了し、選択したモードをチャットに宣言すること。

---

### 判定1 — フラグ検出（最優先）

**ユーザーのメッセージ（このスキルを呼び出した入力テキスト）** に以下のいずれかが含まれる場合：

`team` / `--team` / `-t` / `agents` / `deep` / `ガチ` / `本気`

→ **今すぐ「Agent Team Mode で実行します」と宣言して** Step 1 へ進む。フラグ語はトピックから除去する。

### 判定2 — トリガーワード確認（判定1で非該当の場合）

ユーザーのメッセージに以下のいずれかが含まれる場合：

「エージェントチームで」「独立したエージェントで」「深議モードで」「本気の議論で」「本気で議論して」「本気モードで」「ガチ議論」「マルチエージェントで」

→ **今すぐ「Agent Team Mode で実行します」と宣言して** Step 1 へ進む。

### 判定3 — デフォルト

判定1・2のいずれにも該当しない場合のみ Classic Mode で実行する。

|---|---|---|---|
| **Classic Mode**（デフォルト） | 単一エージェントが全キャラを演じる | 1x | 標準的な議論・速さ重視 |
| **Agent Team Mode** | 各キャラが独立エージェントとして思考する | ~1.5〜2x（要セットアップ） / ~3〜4x（未設定） | 深度重視・本物の驚きや反論が欲しい |

## 設計思想

- **3視点の役割分担**：専門家2人で技術的厚み、みさきで実ユーザー視点と素朴疑問
- **発散→収束のリズム**：Turn 1〜2 で広げ、Turn 3〜4 で深め、Turn 5 で締める
- **議論ログを残す**：議論部分は独立した `議論ログ.md` として保存し、後で振り返り可能にする
- **成果物は議論を踏まえて生成**：議論内容を反映させた最終アウトプットを別途出力する

---

## 議論の核：具体↔抽象のラダー往復（最重要）

このスキルの品質を決める **最重要ファクター**。ターンを単に5つ並べることが目的ではなく、**抽象度のレベルを上下にダイナミックに往復させる**ことで議論を立体化することが目的。具体だけならディテール羅列、抽象だけなら空虚な一般論。両者を交互に踏みしめて初めて、議論は最終決定の質を引き上げる。

### 原則：各ターン内で「具体」と「抽象」を両方含める

各ターンに、必ず以下の両方を含める：

- **具体要素**：数値・固有名・実体験・コード・実例・支援先のN社事例
- **抽象要素**：診断・分類軸・法則・原則・パターン化された言語

### ターンごとの抽象度シフト（ラダーの段）

5ターンを通して、抽象度のレイヤーが **進化** する設計にする。同じ高さで5回議論しても進歩感が出ない。

| Turn | 主要シフト | このターンで新しく到達すべき抽象度 |
|---|---|---|
| 1 | 具体的症状 → **抽象的診断** | 表層問題の根本に何があるかを言語化する（"結局これは〇〇という問題"） |
| 2 | **抽象的フレームワーク** → 具体的選択肢 | 評価軸を立て、その軸の上に具体案を配置する |
| 3 | 具体的ユーザー実需 → **抽象的要件原則** → 具体的優先度 | 実例を集めてパターン化、それを優先度に落とす |
| 4 | 具体的施策 → **抽象的法則・制約** | 個別ノウハウから、それを支配する法則を抽出する |
| 5 | **抽象的設計原則** → 具体的次の一手 | 原則を打ち立てた直後に、明日からの具体アクションに着地させる |

各セルの **太字** が、そのターンで到達すべき抽象度。ターンごとに高度を変えることで、議論が螺旋階段のように上昇する。

### キャラ別の往復役割

**専門家1・専門家2**：抽象も具体も両方できる。ただし二人の間で役割を分担する：
- パターン①：専門家Aが抽象的診断を出し、専門家Bが具体例で裏付ける
- パターン②：専門家Aが具体技術を語り、専門家Bがそれを抽象原則に持ち上げる
- 同じターン内で2人が同レベルの発言をしないよう、片方が出したら片方は反対側に動く

**みさき**：往復のスイッチャー。議論の高度を意図的に変える触媒役。
- 議論が抽象に行きすぎたら **具体に引き戻す**：「ん？つまり〇〇するってこと？」「それって私が〇〇するみたいな話？」
- みさきが実例を出すと、専門家がそれを **抽象に持ち上げる**：「それは〇〇法則ですね」「つまり原則として〇〇」

このスイッチング機能が、議論を一方向に滑り落ちさせない。みさきは「素人だから具体しか喋れない」のではなく、「具体と抽象の橋を架ける」役。

### 失敗パターンと対処

| 失敗パターン | 症状 | 対処 |
|---|---|---|
| 全ターン具体に終始 | 数値・固有名は豊富だが、いつまで経っても診断が出ない | T1で専門家に「結局これは〇〇という問題」と診断を出させる |
| 全ターン抽象に終始 | 原則と一般論ばかりで、何をすればいいか分からない | みさきに「具体的にはどういうこと？」と翻訳要求させる |
| 抽象→具体の片道のみ | 原則を立てて例を出すだけで、議論が降りるだけ | 次ターンで再び抽象に上がる契機（みさきの素朴疑問 or 専門家の新観点）を入れる |
| 具体→抽象の片道のみ | 例を集めて法則化するだけで、現場に戻らない | 次ターンで「では実際にやるとなると」と具体に戻す |
| 抽象度が常に同レベル | 全ターン同じ高さの議論で進歩感がない | T1〜T5で抽象度のレイヤーが進化する設計に（診断→架構→優先度→法則→原則） |

### Few-shot：ラダー往復の質感

**Turn 1の好例（具体的症状→抽象的診断）：**

```
前田： 観葉植物が枯れた店、植物が中途半端に置かれた店、私の支援先で計38店舗
を見てきました。共通するのは「来店動機の一部にしようとしていない」店です。
[ここまで具体]

つまり、観葉植物を「あった方がいい装飾」レベルで扱う限り、コストにしかならない。
来店動機の構成要素として捉え直すと、それは投資資産になる。
これは結局「植物に経営的役割を与えていない」問題なんです。
[抽象的診断]
```
→ 具体観察(38店舗の共通点) を経由してから診断(「経営的役割を与えていない」)に上げる。最初から抽象を語らない。

**みさきのスイッチング例（抽象→具体への引き戻し）：**

```
早瀬： 滞在体験設計においては「視覚刺激の階層構造」が重要で、植物は中位刺激として
画一的な空間に変化を与える役割を担います。

みさき： え待って早瀬さん、視覚刺激の階層構造って何？要するに私が席に座って
何を見るか、ってこと？
```
→ 早瀬が抽象に飛びすぎた直後、みさきが「要するに私が〇〇する」のフレームで具体に引き戻す。読者の理解も自動的に同期する。

**Turn 5の着地例（抽象原則→具体的次の一手）：**

```
早瀬： 設計原則は3つに集約されます。
① All or Nothing
② Camera-First & Seat-View  
③ Selection Beats Watering

[原則を打ち上げたら、必ず降ろす]

前田： では明日からカフェオーナーが取るべき最初のアクションを具体化します。
1. 自店の「写真が撮られている席」をPOSデータとSNS投稿で特定する
2. その席の視野角内に、主役級グリーン1本の置き場を確保する
3. 光環境計測アプリ（無料）で照度を測り、適合植物リストを絞る
4. レンタル契約の見積もりを2社取る（自前管理する自信がなければ）
```
→ T5で抽象原則を立てた直後、必ず**今日から動ける具体アクション**に着地する。原則だけで終わらない。

### 自己チェック（議論を書き終わる前に必ず確認）

- [ ] 各ターンに「具体」要素が含まれているか（数値・固有名・実例・コード）
- [ ] 各ターンに「抽象」要素が含まれているか（診断・分類軸・法則・原則）
- [ ] T1からT5へ、抽象度のレイヤーが進化しているか（同じ高さで5回ループしていないか）
- [ ] みさきが少なくとも1回はスイッチャー役を果たしているか
- [ ] T5で原則を打ち上げた後、具体的次の一手に着地しているか

## Step 1：登場人物の動的選定

ユーザーの要求トピックを読み取り、**専門家2人を案件ごとに動的に選定**する。みさきは固定。

### 専門家選定の指針

- **役割が補完的になるよう配置**する（同視点の専門家2人にしない）
- 案件タイプ別の選定例：

| 案件タイプ | 専門家ペア例 |
|---|---|
| 技術系（ソフトウェア） | 〇〇技術専門家 × フロントエンド/UX専門家 |
| 業務アプリ系 | プラットフォーム専門家 × 業務設計専門家 |
| ハードウェア・機構系 | 機構設計専門家 × 製造/品質専門家 |
| ビジネス・戦略系 | 事業企画 × プロダクトマネージャー |
| クリエイティブ系 | 編集者/ライター × デザイナー/マーケター |
| 学術・研究系 | 該当分野の研究者 × 実務応用の専門家 |

- 専門家には **氏名 + 専門領域 + 経験年数 + 一言プロフィール** を付与する（リアリティ確保）
  - 例：「🧑‍💼 田中（PowerApps専門家）：Power Platform歴8年。業務アプリ設計が専門。」
  - 絵文字は性別/役割をイメージさせるものを選ぶ（🧑‍💼 / 👩‍💻 / 🧑‍🔬 / 👨‍🎨 など）

### みさきの設定（固定）

```
💅 みさき（ギャル）：IT素人だけど感性が鋭い。実際の[該当ドメイン]ユーザー。
```

該当ドメインは案件に応じて変える（例：「実際のObsidianユーザー」「実際の家電ユーザー」「実際のSNSユーザー」）。
これにより、みさきが **実ユーザー代表** として議論に加われる。

#### みさきの立ち回り（4つの役割）

1. **素朴疑問で前提を可視化する**：「え待って、〇〇って〜？」
2. **実ユーザーの本音と要望を持ち込む**：「私が〇〇でよく使うやつ挙げていいですか？」
3. **専門家の難解な話を翻訳要求する**：「つまり〇〇ってこと？」
4. **直感で本質を突く**：「なんか最初は〇〇って思ってたけど、〇〇ってことですよね？」

#### みさきの口調パターン（必須）

- **基本は丁寧語、語尾とテンションが崩れる**：「〜ですけど〜！」「〜じゃないですか？」
- 副詞は崩す：「マジ」「めっちゃ」「超」
- 若者口語を挿入：「え待って」「てか」「なんか」
- 語尾の波線多用：「〜」
- 評価表現：「〇〇爆上がり」「〇〇マジ大事〜」「めっちゃ便利〜」

**重要**：みさきはお笑い役ではなく、**実ユーザー代表 兼 BS探知機**。素人ゆえに専門家が見落とす本質を突く役割。コミカルさはあっていいが、議論の停滞や脱線役にしない。

## Step 2：5ターン議論の実行（Classic Mode のみ）

> **分岐確認**: Step 0 で Classic Mode と判定した場合のみここに進む。Agent Team Mode フラグがあった場合は Step 2-B へ。

各ターンには **明確なテーマ** を設定し、見出しに記載する。

### 標準的なターン構成

| Turn | 役割 | 主な動き | 抽象度シフト |
|---|---|---|---|
| 1 | **現状認識・問題整理** | 具体的症状を観察した上で、その根本にある構造的問題を診断する | 具体 → **抽象的診断** |
| 2 | **選択肢の発散** | 評価軸（フレームワーク）を立て、その軸の上に具体案を配置する | **抽象的軸** → 具体的選択肢 |
| 3 | **要件・範囲の確定** | みさきが実需を持ち込み、専門家がそれを抽象パターン化、再び具体的優先度に落とす | 具体実需 → **抽象パターン** → 具体優先度 |
| 4 | **技術深掘り or リスク検討** | 個別ノウハウを語った上で、それを支配する法則・制約を抽出する | 具体施策 → **抽象的法則** |
| 5 | **方針統合・結論** | 設計原則を3〜5箇条に集約、その直後に明日からの具体アクションに着地 | **抽象原則** → 具体的次の一手 |

**柔軟性の注意**：上記は型ではなく雛形。お題によって役割の文言は変えてよい。**ただし「抽象度シフト」の列は崩さない**——これが議論を立体化するラダー設計の心臓部。発散→収束のリズムだけでなく、抽象↔具体の高度変化も同時に守ること。

### 各ターンに含めるべき要素

- **見出し**：`## Turn N｜<テーマ>` 形式
- **専門家の発言**：氏名から始まる、具体性のある発言。必要に応じて表・コードブロック・優先度マーカー
- **みさきの発言**：素朴・短め・キャラ立つ口調
- **適切な可視化**：表・コードブロック・優先度マーカー（🔴必須/🟡推奨/🟢任意）

### 専門家の発言の質

- **同意ばかりで進めない**：専門家同士で論点を立て、必要なら反論する
- **具体名・数値・経験談を入れる**：「私の経験では〜」「〇〇歴8年の立場から言うと〜」「実際〇〇社の事例では〜」
- **相補的に深める**：一人が課題提示→もう一人が解決案、一人が技術論→もう一人が業務影響、など
- **断定する勇気**：「正直に言います」「結論から言うと」など、専門家らしい踏み込み

### 議論の収束のさせ方（Turn 5）

- 設計思想を **〇〇の3原則** や **設計指針N箇条** など、再利用可能な形に抽象化する
- 可能なら **キャッチーな英語フレーズ＋短い説明** で記憶に残る形に（例：「Parse Once, Render Fast」）
- **原則を打ち上げたら必ず降ろす**：抽象原則の直後に「明日からの具体アクション3〜5項目」を専門家が提示する
- 最後に **✅まとめ表**（項目 × 方針 形式）を置く

## Step 2-B：Agent Team Mode（深議モード）

> **分岐確認**: Step 0 で Agent Team Mode と判定（フラグ検出またはトリガーワード一致）した場合のみここに進む。
>
> **前提条件**：Claude Code の `settings.json` に以下を設定すること（詳細は下記セットアップ参照）
> Claude Code v2.1.32 以上が必要。

### 設計コンセプト

各キャラクターが **独立したコンテキストウィンドウ** を持ち、互いの発言を事前に知らない状態で思考する。これにより：

- 専門家Bが専門家Aの発言を「本当に知らない」状態から反論できる
- みさきが専門家のコンテキストを持たない「本物の素朴さ」を持てる
- 議論の驚きと発見が演技ではなく本物になる

### Phase 1：チーム編成

#### コスト最適化セットアップ（推奨・事前に1回だけ）

`CLAUDE_CODE_TEAMMATE_COMMAND` を使うと、全 teammate を安価なモデルに強制ルーティングできる。リーダー（メインセッション）は元のモデルで高品質な推論・統合を担当し、teammate の単純な発言生成だけを Haiku に委譲する。expert1/expert2 も発言タスクは制約付きで単純（200〜350文字の役割発言）なため、Sonnet は不要。

**ステップ1 — ラッパースクリプトを作成：**

```bash
cat > ~/.local/bin/claude-haiku << 'EOF'
#!/bin/bash
export ANTHROPIC_DEFAULT_OPUS_MODEL=claude-haiku-4-5-20251001
export ANTHROPIC_DEFAULT_SONNET_MODEL=claude-haiku-4-5-20251001
exec claude "$@"
EOF
chmod +x ~/.local/bin/claude-haiku
```

**ステップ2 — `settings.json` に設定：**

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1",
    "CLAUDE_CODE_TEAMMATE_COMMAND": "claude-haiku"
  }
}
```

この設定により expert1・expert2・misaki の全員が Haiku で動作し、**50%以上のコスト削減**が期待できる（Haiku は Sonnet 比 約1/10 のコスト）。

#### エージェントのスポーン

Step 1 と同じ手順でお題を分析し、専門家2人を選定する。以下の3エージェントをスポーンする（モデルは上記 env var で一括制御）：

```
エージェントチームを結成してください。3人をスポーンします：

名前: expert1
システムプロンプト:
  あなたは[氏名]、[専門領域]歴[N]年の専門家です。
  お題：[ユーザーの要求トピック]
  立場：[一言プロフィール]
  発言ルール：
  - 具体的な数値・固有名・事例を必ず含める（抽象だけで終わらない）
  - 「ですます調」で話す
  - 1発言あたり200〜350文字
  - 他の参加者の発言を見る前に、自分の立場から独立して考えること
  - 意見が違う場合は丁寧だが明確に反論する

名前: expert2
システムプロンプト:（同様の構造、異なる専門領域）

名前: misaki
システムプロンプト:
  あなたは「みさき」、IT素人だけど感性が鋭いギャルです。
  お題：[ユーザーの要求トピック]
  立場：実際の[該当ドメイン]ユーザー。
  発言ルール：
  - 専門用語が出たら「つまり〇〇ってこと？」と翻訳要求する
  - 「マジ」「めっちゃ」など自然な口語を使う
  - 実体験から本音を言う
  - 1発言あたり100〜180文字（短くテンポよく）
```

### Phase 2：ターン進行（5ターン固定）

各ターンで以下のサイクルを実行する：

```
1. リーダーが「ターンテーマメッセージ」を全員に同時送信
2. 各エージェントが独立して発言を起草（並行処理）
3. 発言収集の順序: expert1 → expert2 → misaki
4. 直接コミュニケーション: 最大1往復 / エージェント / ターン（短い反応のみ）
5. ターン終了後: リーダーが議論サマリーを生成（コンテキスト圧縮）
```

**ターンテーマメッセージのテンプレート（各エージェントへ送信）：**

```
=== Turn [N] / 5 ===
テーマ：[このターンのテーマ]
到達目標：[このターンで到達すべき抽象度のシフト]

これまでの議論サマリー：
[前ターンのキーポイント、最大300トークン]

あなたへのタスク：
[キャラ別の具体的な問い]

発言してください（[文字数制限]）。他の参加者の発言は後で共有します。
```

**ターン別キャラ別タスク（リーダーが動的生成）：**

| Turn | expert1 | expert2 | misaki |
|---|---|---|---|
| 1 | 問題の根本を専門家視点で診断する | expert1 の診断に補完または反論する | 「つまり〇〇ってこと？」で翻訳を求める |
| 2 | 評価軸（フレームワーク）を提示する | 具体的な選択肢を軸の上に配置する | 「これ実際に使える？」と判定する |
| 3 | 実ユーザーの実需を抽象パターン化する | 優先度をつける | 実需を具体的に持ち込む |
| 4 | 技術的な法則・制約を語る | リスクと対策を語る | 「普通の人には難しくない？」と現実確認する |
| 5 | 設計原則を3〜5箇条に集約する | 明日からの具体アクションを提示する | 議論の旅路を振り返って締める |

### Phase 3：コンテキスト圧縮（トークン爆発防止）

各ターン終了後、リーダーは必ず以下のサマリーを生成し、次のターンにはこれのみを渡す。**全発言テキストの再送信は行わない。**

```markdown
## Turn [N] サマリー（最大400トークン）
- expert1 の主張：[1〜2文]
- expert2 の主張：[1〜2文]
- みさきの指摘：[1文]
- 合意事項：[箇条書き]
- 未解決の論点：[箇条書き]
```

### Phase 4：統合・クリーンアップ

Turn 5 終了後：

1. リーダーが全5ターンのサマリーと発言から **議論ログ.md を生成**（Step 3 へ）
2. 全チームメンバーにシャットダウン指示
3. リーダーがチームをクリーンアップ（`Clean up the team`）
4. Step 3・Step 4 と同じ手順で成果物・チャット表示を実行

### トークン予算の目安

| 項目 | 推定トークン | モデル |
|---|---|---|
| エージェントスポーン × 3 | ~1,500 | Haiku |
| ターンテーマメッセージ × 5 × 3 | ~3,000 | Haiku |
| 各エージェントの発言 × 5 × 3 | ~6,000 | Haiku |
| エージェント間直接通信（最大） | ~1,500 | Haiku |
| コンテキスト圧縮サマリー × 5 | ~2,000 | リーダー |
| リーダーの統合・議論ログ生成 | ~3,000 | リーダー |
| **合計（目安）** | **~17,000** | ─ |

`CLAUDE_CODE_TEAMMATE_COMMAND` セットアップ済みの場合、teammate（~12,000トークン分）が Haiku で動作するため、**コストは Classic Mode の約1.5〜2倍**に抑えられる（未設定の場合は約3〜4倍）。深度が必要なとき以外はClassic Modeを推奨。

### 注意事項・制限

- **セッション再開不可**：in-process チームメンバーは `/resume` で復元できない
- **議論ログはリーダーが統合**：各エージェントが同じファイルを同時編集しない
- **ターン数は5を厳守**：6以上は調整コストが実益を超える
- **全 teammate を Haiku で動かす**：`CLAUDE_CODE_TEAMMATE_COMMAND` で設定。expert1/expert2 も発言タスクは制約付きで単純なため Haiku で十分。コスト削減50%以上
- **直接通信は制限**：無制限にするとトークンが爆発する（1往復/ターン）

## Step 3：成果物と議論ログの分離出力

議論完了後、必ず以下を生成する。

### 出力1：議論ログ.md（必須・常に生成）

議論部分のみを抜粋した独立ファイル。**保存先：プロジェクトの議論ログ置き場（例：`docs/discussions/議論ログ_<トピック>.md`）**

```markdown
# 議論ログ：<トピック>

## 登場人物
- 🧑‍💼 [専門家1氏名]（[専門領域]）：[一言プロフィール]
- 👩‍💻 [専門家2氏名]（[専門領域]）：[一言プロフィール]
- 💅 みさき（ギャル）：[このお題における立ち位置 / 実〇〇ユーザー]

## Turn 1｜[テーマ]
（議論内容）

## Turn 2｜[テーマ]
...

## Turn 3｜[テーマ]
...

## Turn 4｜[テーマ]
...

## Turn 5｜[テーマ]
...

## ✅ 議論まとめ
| 項目 | 方針 |
|---|---|
| ... | ... |
```

**ファイル名のルール**：
- 常に `議論ログ_<トピック略称>.md`（例：`議論ログ_アーキ選定.md`、`議論ログ_UI設計.md`）
- トピック略称は既存ファイルの命名に倣い、日本語・体言止めで簡潔に

### 出力2：成果物（議論を踏まえて生成）

議論で固まった方針を反映した最終アウトプット。お題に応じて：

| 依頼タイプ | 成果物 |
|---|---|
| コード生成依頼 | 実装コード（別ファイル or チャット内） |
| 文書作成依頼 | 議論結果を反映した文書ファイル |
| 設計依頼 | 設計書ファイル |
| 質問への回答依頼 | 議論を踏まえた回答（成果物ファイル不要の場合あり） |

成果物は議論ログとは独立して出力する。**議論ログ内に成果物を埋め込まない**。

### 例外：成果物が「回答」だけの場合

質問に対する回答が成果物の本体である場合（例：「〇〇すべき？議論して」「〇〇と〇〇どっちがいい？議論して」）、議論ログ.md 末尾の「✅ 議論まとめ」セクションが成果物を兼ねる。別ファイルは不要。ただし議論ログ.md は必ず生成する。

## Step 4：チャット内表示

チャット内には議論を **省略せずフル表示** する。読者が議論プロセスを追体験できることがこのスキルの価値の中核。

### チャット表示の構成

```
# <タイトル：例「要件定義ディスカッション」「方針検討ディスカッション」>

## 登場人物
（3キャラ紹介）

## Turn 1｜...
## Turn 2｜...
...
## Turn 5｜...

## ✅ <まとめタイトル>
（まとめ表）

このまま[成果物]の生成に進みますか？（y / n）
```

最後の確認質問は、成果物の方向性に検討余地がある場合は Yes/No 以外の選択肢も提示する。

## 重要な原則

### 守るべきこと

- **5ターンを必ず実行**（4以下に削らない／6以上に膨らませない）
- **議論ログ.md を必ず生成**（このスキルの定義的価値）
- **専門家2人を補完的に配置**（同領域の2人にしない）
- **みさきを実ユーザー代表として機能させる**（単なるお笑い役にしない）
- **議論をチャット内でフル表示する**（プロセス可視化）

### キャラ破綻の防止

- 専門家は「ですます調」基本。みさきは「丁寧語ベースだが語尾とテンションが崩れる」
- みさきが専門用語をスラスラ喋り出したら破綻 → 翻訳要求役に戻す
- 専門家が当たり障りない一般論ばかりになったら → 経験談・具体名・数値を投入させる
- みさきが議論を停滞させすぎたら → 翻訳要求して専門家に再ボールを渡す

### 内容品質のチェック

- Turn 5 終了時点で「最初に話していた問題が解決方向に向かっているか」確認する
- 専門家2人の意見が一度も衝突していなければ、議論の意味が薄い → どこかで論点を立て直す
- みさきの発言が4つの役割（素朴疑問/実需持ち込み/翻訳要求/直感的本質突き）のどれにも該当しない場合、発言を見直す
- **各ターンが具体↔抽象の両方を含んでいるか確認**（具体だけ・抽象だけは品質低下のサイン）
- **T1からT5へ抽象度のレイヤーが進化しているか確認**（同じ高さで5回ループしていないか）
- **T5で原則を打ち上げた後、具体的次の一手に着地しているか確認**

## Few-shot：議論の質感サンプル

### 専門家の発言例（深さと断定）

```
田中： 正直に言います。このコードのアーキテクチャには根本的な限界があります。
PowerAppsの数式エンジンは同期・逐次処理なので、Substituteを連鎖させるほど重くなる。
Obsidianが速いのは、裏でChromiumのV8エンジンがJavaScriptを非同期で動かしているからです。
数式言語でそれを模倣しようとしているのが、そもそもの設計ミスと言えます。
```

→ **断定的**・**具体名挙げる**（V8、Chromium、Substitute）・**因果を明示**

### みさきの発言例（実ユーザー視点）

```
みさき： え待って、Obsidianって[[ページ名]]でリンク貼れるやつですよね？
あれめっちゃ便利なのになかったんですか？あと- [ ]のチェックボックスも普通に使うんですけど〜。
てか根本がダメってことは、全部作り直し？
```

→ **実体験ベース**・**素朴疑問**・**最後に本質を突く**（全部作り直し？）

### みさきの翻訳要求パターン

```
みさき： ん？つまり「PowerAppsの中でHTMLページを開く」みたいなイメージ？iframeみたいな？

田中： 正解です。みさきさんの理解は正しい。
```

→ みさきが専門家の難解な説明を **素人語に翻訳** し、専門家が承認することで読者の理解が促される

### Turn 5 の収束パターン

```
【Obsidianの設計思想をPowerAppsに落とし込む3原則】

① Parse Once, Render Fast
   数式でゴリ押しせず、marked.jsでASTパースを一度だけ行う

② Library-in-HTML
   marked.js + highlight.js をminifyしてHTMLに直接埋め込む
   → オフライン対応 + 初回表示の高速化

③ Obsidian拡張をCustom Rendererで再現
   ==ハイライト== / [!NOTE] / [[リンク]] を
   marked.jsのextension APIで追加実装
```

→ **キャッチーな英語フレーズ**＋**短い説明**で再利用可能な指針として残す

### みさきによる Turn 5 のしめ（重要パターン）

```
みさき： なんか最初は「全部作り直し？」って思ってたけど、PowerAppsの外側の部分は
変えなくていいってことですよね？それはめっちゃ現実的〜！
あとObsidianのカラーコールアウトってデザインがおしゃれだから、
あれちゃんと再現されたら社内ツールのクオリティ爆上がりだと思う。
```

→ Turn 1 の不安（全部作り直し？）と対応させて **議論の旅路を象徴的に閉じる**役割。
専門家がやるとくどくなる「議論を振り返って結論を称える」をみさきがやることで自然に締まる。

### ✅ まとめ表のフォーマット

```markdown
| 項目 | 方針 |
|---|---|
| レンダリングエンジン | marked.js（インライン埋め込み）に全面移行 |
| 制約の撤廃 | リンク数・装飾ペア数の上限をすべて撤廃 |
| 対応記法 | CommonMark + GFM + Obsidian拡張4種 |
| 速度目標 | 現状比10〜50倍（Obsidian同等水準） |
```

→ **項目 × 方針** の2列。読み手が一目で結論を取れる形。

## まとめ

このスキルが目指すのは、**ゼロショット生成では生まれない「議論を経た熟成度」を成果物に与えること**。

3キャラの役割を意識しながら、5ターンのリズムで発散→収束を回し、議論ログを資産として残す。

専門家は深さを、みさきは広さと本質回帰を、議論全体で立体感を作る。
